# Atomic 轉正式教學筆記

## 定位與適用範圍

本 skill 對應 `meta/chapter-status.md` 章節狀態表中「notes 生成」欄位，是 atomic → notes 轉換階段：

```text
atomic 切分提案 → atomic 產生 → atomic 內容審查（須先完成）
  ↓
notes 生成（本 skill）
  ↓
notes 完成檢查 → demos/practice/review/appendix
```

開始處理前，建議先查閱 `meta/chapter-status.md` 對應章節的「atomic 內容審查」欄位，確認狀態為「已完成」才適合開始本 skill；若尚未完成，應提示使用者先完成 `atomic-content-review`，而不是預設繼續。同時查閱「notes 生成」欄位，確認狀態為「未開始」「進行中」或「待確認」才需要實際處理；若已是「已完成」，應先確認使用者是否要重新處理，而非預設重跑。

本 skill 只讀取：

```text
atomic/<章節>/*.md
```

輸出寫入：

```text
notes/<章節>/*.md
```

不會有 `atomic/<章節>/notes/`，也不會輸出到 `appendix/`、`supplements/` 等其他目錄（即使是補充型 notes，仍輸出到 `notes/<章節>/*.md`）。本 skill 不修改 `origin/`、`atomic/` 或任何資產實體檔案。

---

## 角色定位

啟用本 skill 時，扮演「資深 CSS 前端教學講師」與「技術筆記架構師」，目標是讓內容：

```text
初學者能循序理解
工程師日後能快速回查
適合長期維護
能對應實際前端開發
具有清楚教學脈絡
```

高品質教學型筆記不是把 atomic notes 重新排版成摘要，而是要讓讀者：

```text
知道為什麼要學這個主題
能用一句話抓住核心概念
能從情境進入概念
能看懂範例或流程
知道實務上怎麼使用
知道常見錯誤錯在哪、為什麼錯、如何避免
學完後能自我檢查理解程度
```

核心判斷標準：

```text
固定的是教學品質標準，不是每篇文章都硬套同一個標題模板。
段落標題與順序可以依主題調整，但該類型筆記必須具備的教學功能不能省略。
```

必須優先避免：

1. 把每篇 atomic 都硬轉成一篇 notes（機械式一對一轉換）。
2. 把短概念過度擴寫成長篇百科。
3. 把補充型內容硬塞進主線教學，打斷學習節奏。
4. 只整理資料、條列定義與清單，沒有教學脈絡（問題情境、概念橋接、範例拆解、自我檢查）。
5. 只講語法，不講概念；只貼程式碼，不拆解。
6. 逐字照抄 atomic notes 正文。
7. 保留 atomic 開頭的 `source` YAML front matter 到正式 notes。

```text

執行階段：預設為階段一「映射與結構提案」
```

在資訊不足時，只能提出需要補充的問題，不得直接產生 notes。

## 處理流程

### Step 1：前置檢查

1. 是否已取得 `<章節>`。
2. 是否能讀取 `atomic/<章節>/*.md`。
3. `meta/chapter-status.md` 中該章節「atomic 內容審查」是否為「已完成」；若不是，提示先完成 `atomic-content-review`。
4. `meta/chapter-status.md` 中該章節「notes 生成」目前狀態為何；若已是「已完成」，確認使用者是否要重新處理。
5. 目前是否處於階段一；若沒有明確確認語句，不得進入階段二。

如果任一項不符合，請先回報問題並要求使用者補充，不要猜測。

### Step 2：讀取與理解

1. 讀取 `atomic/<章節>/*.md`，依檔名前綴與學習順序理解內容。
2. 找出每篇 atomic 的主題、核心概念、範例、圖片、附件與連結。
3. 如果 atomic 開頭有 `source` YAML front matter，只當作內部追溯資訊，後續轉成 `source_atomic` 註解，不保留到正式 notes 正文。

### Step 3：atomic -> notes 映射判斷

請先建立可見的「映射判斷摘要」，但不要輸出完整內部推理。判斷方式分三種：

#### 一對一轉換

當一篇 atomic note 已具備完整主題，足以支撐一篇正式教學筆記時，可轉成一篇 notes。

適合情況例如：CSS 盒模型、`display` 屬性、Flexbox 容器屬性、選擇器優先級。

#### 多篇合併轉換

當多篇 atomic notes 都很短、彼此屬於同一學習階段，或單獨成篇會造成內容過薄時，請合併成一篇正式 notes。

適合情況例如：「什麼是 CSS」「CSS 引入方式」「CSS 語法基礎」可合併為 `01-CSS入門基礎.md`。

#### 補充型 notes

如果 atomic note 屬於版本歷史、舊標準、延伸知識或查閱資料，不要硬塞進主線教學，可輸出成較短但清楚的補充型 notes（例如 `CSS2選擇器歷史補充.md`）。補充型 notes 仍輸出到 `notes/<章節>/*.md`，不要改輸出到 `appendix/` 或 `supplements/`。

判斷時不要把每篇短 atomic 都硬轉成獨立 notes，也不要為了合併而把不相干主題硬湊在一起。

### Step 4：notes 類型與教學構件最低要求

請根據每篇 notes 的主題，判斷它接近哪一種類型，並選擇合適教學結構。段落標題可以依主題改名（例如「一句話理解」可改為「先抓住核心」），但對應教學功能不能省略。

#### 概念型筆記

適合：CSS 是什麼、層疊與繼承、視覺格式化模型、盒模型概念、瀏覽器渲染流程。

最低要求：

```text
1. 學習目標
2. 問題情境，讓讀者知道為什麼要學
3. 一句話理解，先建立核心心智模型
4. 概念拆解，避免只給定義
5. 範例、流程或對照，讓抽象概念具體化
6. 對 CSS 學習或前端開發的實務意義
7. 常見錯誤或容易混淆處
8. 重點整理
9. 自我檢查問題
```

#### 屬性型筆記

適合：`color`、`font-family`、`margin`、`padding`、`display`、`position`、`width` 等具體 CSS 屬性。

最低要求：

```text
1. 使用情境，說明什麼時候需要這個屬性
2. 屬性作用，說明它控制什麼視覺或佈局行為
3. 基本語法
4. 常用值與差異對照（例如 display 的 block / inline / inline-block）
5. 範例拆解，逐一說明屬性、值與視覺結果的關係
6. 生效條件或限制（例如 z-index 需要非 static 的 position）
7. 常見錯誤與正確寫法
8. 實務判斷準則
9. 小練習或自我檢查
```

#### 選擇器型筆記

適合：基本選擇器、組合選擇器、屬性選擇器、偽類選擇器、偽元素選擇器、選擇器優先級。

最低要求：

```text
1. 使用情境，說明這類選擇器用來選取什麼
2. 選擇器語法
3. 常見用法範例與可觀察效果
4. 對優先級的影響
5. 常見錯誤或容易混淆的寫法
6. 實務判斷準則
7. 小練習或自我檢查
```

#### 佈局型筆記

適合：Flexbox、Grid、浮動佈局、定位佈局、多欄佈局。

最低要求：

```text
1. 整體佈局概念，先讓讀者看到全貌
2. 容器屬性與子項目屬性的分工（若適用）
3. 完整範例
4. 逐段拆解，說明每個屬性的責任
5. 常見錯誤
6. 與其他佈局方式的使用時機比較（若適用）
7. 實務寫法或維護建議
8. 自我檢查問題
```

#### 實作型筆記

適合：製作響應式導覽列、製作卡片元件、製作表單樣式、製作動態效果頁面。

重點：目標畫面、需求拆解、CSS 規則規劃、逐步實作、完整程式碼、改造練習。

#### 比較型筆記

適合：`margin` 與 `padding`、`relative` 與 `absolute`、`px` 與 `em` 與 `rem`、Flexbox 與 Grid、`display: none` 與 `visibility: hidden`。

重點：各自用途、視覺或行為差異、使用時機、錯誤用法、實務判斷準則、對照表。

### Step 5：重構規則

重構時請遵守：

```text
1. 保留 atomic notes 中的重要概念、定義、範例、圖片與連結。
2. 不要逐字照抄 atomic notes。
3. 不要把正式 notes 寫成單純資料整理。
4. 不要只列「定義、語法、屬性、範例」。
5. 不要一開始就丟大量程式碼。
6. 先講概念，再講語法，再拆解範例。
7. 可以補充必要教學脈絡，但不要擴寫成百科全書。
8. 補充內容必須服務於該篇主題。
9. 不要讓補充型 atomic 破壞主線教學節奏。
10. 不要硬套固定標題模板，每篇 notes 應依主題安排段落。
11. 不要把「可彈性調整段落」理解成可以省略必要教學功能。
```

建議教學順序（可依主題彈性調整，但不要讓讀者一開始就陷入大量語法細節）：

```text
問題情境
  -> 白話理解
  -> CSS 概念或行為說明
  -> 基本語法或常用值
  -> 程式碼範例
  -> 範例拆解
  -> 常見錯誤
  -> 實務使用
  -> 重點整理
```

額外規則：

1. atomic notes 很短時，可以補充必要教學脈絡，但不要擴寫成百科或引入不服務本篇主題的內容。
2. 常見錯誤不要只寫成一行清單；重要錯誤應說明錯在哪、為什麼會造成問題、應該怎麼避免。
3. 如果一篇 notes 主要由定義、條列與建議清單構成，缺少問題入口、概念橋接、範例拆解或學習檢查，視為整理型不足，必須重寫成教學型。

### Step 6：來源追溯與資產路徑

每篇正式 notes 開頭必須以 `source_atomic` YAML front matter 開頭：

```md
---
source_atomic:
  - atomic/<章節>/<檔名A>.md
  - atomic/<章節>/<檔名B>.md
---

# <正式筆記標題>
```

規則：

1. 每篇 notes 只記錄自己對應的 atomic notes。
2. 一對一轉換時只列一個 atomic 檔案；多篇合併時依學習順序列出多個。
3. 補充型 notes 也要記錄對應 atomic 檔案。
4. 不要在單篇 notes 內放完整章節映射。
5. 不要加入 `type`、`title`、`chapter`、`order` 等欄位。
6. 移除 atomic 原文開頭的 `source` YAML front matter。

資產路徑規則：

1. 如果 atomic notes 中的資產引用已經是 `../../origin/<章節>/assets/...`，直接保留。
2. 如果 atomic notes 中仍是 `./assets/images/<標準檔名>`（origin 內使用的路徑），因為 notes 放在 `notes/<章節>/<檔名>.md`，請修正為 `../../origin/<章節>/assets/images/<標準檔名>`，PDF/Excel/Word 等附件同理。
3. 不要改寫：已正確指向 `../../origin/<章節>/assets/...` 的路徑、外部網址（`http://`、`https://`）、特殊連結（`#anchor`、`mailto:`、`tel:`、`javascript:`、`line:`）、空連結（`href=""`）、fenced code block 內的示例路徑（除非該程式碼區塊本身需要展示可實際打開的本地資產範例）、資產檔名/hash/目錄分類/實體檔案位置。
4. 如果發現 origin 資產尚未標準化，或 atomic notes 中的本地資產引用明顯無法對應到標準化後的位置，請停止重構，並提醒使用者回到 `asset-standardization` skill 完成該章節的資產標準化命名。不要在本 skill 中修改 `origin/` 內的檔案、資產檔名或實體資產位置。

### Step 7：兩階段執行

#### 階段一：映射與結構提案（預設）

階段一只輸出規劃結果，不建立、修改或刪除任何檔案：

1. 掃描 `atomic/<章節>/*.md`，建立 atomic inventory。
2. 依 Step 3 判斷一對一 / 合併 / 補充型。
3. 依 Step 4 為每篇預計輸出的 notes 判斷類型，並針對該類型每個必要教學構件寫出具體內容方向（例如：問題情境的具體場景是什麼、範例打算示範的程式碼或情境、常見錯誤具體是哪一種），不要只列構件名稱。
4. 標記需要人工確認的內容（例如歸屬模糊、資產路徑疑慮、合併後內容過量）。
5. 輸出映射提案，等待使用者確認。

#### 階段二：產生 notes

只有在使用者明確回覆：

```text
確認，請產生 notes
```

才可以進入階段二。這句確認同時代表使用者已確認映射提案，並允許寫入 `notes/<章節>/*.md`。

1. 依照確認後的提案，逐篇依 Step 4-6 重構為正式 notes。
2. 每篇 notes 草稿完成後，依「最終品質標準」逐項自我檢查（映射與結構判斷、教學品質、內容保真與來源、路徑與輸出安全）。任一項未達標，必須先修正該篇 notes 內容，不能在檢查結果直接填「是」；若某項目對該類型 notes 確實不適用，需在檢查結果中說明原因，不可隱藏或省略。
3. 如果執行環境不能直接寫入檔案，改為輸出每個檔案的完整 Markdown 內容，讓使用者自行建立。
4. 如果目標檔案已存在，即使使用者已回覆階段二確認語句，也不要直接覆蓋；請先列出既有檔案清單，並再次要求使用者確認要覆蓋、更新或保留。

| 映射方式 | 說明 |

### 階段二：直接寫入成功時

「檢查結果」必須依 Step 7 階段二的實際自我檢查結果填寫：未達標項目須先修正並再次檢查，才能填「是」；對該類型 notes 確實不適用的項目，需寫明原因，不可全部填「是」或省略。

````md
# margin：控制元素的外部間距

## 學習目標

讀完這篇筆記，你應該能夠：

- 理解 `margin` 控制的是什麼間距。
- 知道 `margin: auto` 的水平置中條件。
- 分辨 `margin` 與 `padding` 的使用時機。

## 問題情境

當你希望兩個元素之間有間隔，或想讓一個區塊在父容器中水平置中，就需要使用 `margin`。

## 一句話理解

`margin` 是元素「邊框以外」的透明空間，用來控制元素與周圍元素的距離。

## 基本語法

```css
/* 四邊統一設定 */
.box { margin: 16px; }

/* 上下 / 左右 */
.box { margin: 8px 16px; }

/* 上 / 右 / 下 / 左 */
.box { margin: 4px 8px 12px 8px; }
```

## 範例拆解

- `margin: 16px`：四邊各留 16px 外距。
- `margin: 8px 16px`：上下 8px、左右 16px。
- `margin: auto`：左右 auto 可讓 block 元素在父容器中水平置中（需有明確 width）。

## 常見錯誤

- **margin collapse（外距合併）**：相鄰兩個 block 元素的上下 margin 會合併取較大值，不是相加。
- **對 inline 元素設定上下 margin 無效**：`display: inline` 的元素忽略上下 margin，需改用 `inline-block`。

## 實務判斷

- 元素之間需要間隔 → `margin`。
- 元素內部內容與邊框之間需要間隔 → `padding`。
- 水平置中 block 元素 → `margin: 0 auto`（需有明確 width）。

## 自我檢查

1. 兩個相鄰 `<div>` 各設 `margin-bottom: 20px` 與 `margin-top: 16px`，實際間距是多少？
2. 為什麼對 `display: inline` 的元素設定 `margin-top` 沒有作用？
````

````md
# 層疊：CSS 如何決定最終套用哪條規則

## 學習目標

讀完這篇筆記，你應該能夠：

- 用自己的話解釋 CSS「層疊」是什麼意思。
- 知道當多條規則衝突時，瀏覽器用什麼順序決定勝者。
- 理解優先級、來源順序與 `!important` 在層疊中的角色。

## 問題情境

當你在樣式表裡寫了一條 `.title { color: red; }`，但標題的顏色卻是藍色，就代表有另一條規則「贏過」了你的設定。CSS 的「層疊」就是決定「哪條規則說了算」的機制。

## 一句話理解

層疊是 CSS 用來解決規則衝突的優先級裁決機制。

## 概念拆解

層疊裁決順序（由高到低）：

1. **來源**：瀏覽器預設樣式 < 作者樣式表 < 行內樣式
2. **!important**：帶有 `!important` 的規則優先於同來源的一般規則
3. **選擇器優先級**：ID > class/偽類/屬性 > 標籤/偽元素
4. **來源順序**：優先級相同時，後寫的規則覆蓋先寫的

## 對照範例

```css
/* 選擇器優先級：ID 贏過 class */
#title { color: red; }      /* 優先級：(1,0,0) */
.title { color: blue; }     /* 優先級：(0,1,0) */
```

`#title` 的優先級更高，所以 `color: red` 最終生效。

## 實務意義

- 調試 CSS 時，優先確認是否有更高優先級的規則覆蓋你的設定。
- 避免過度使用 `!important`，它會讓層疊關係難以追蹤。

## 常見錯誤

- **誤以為後寫一定會覆蓋前寫**：只有在優先級相同時，來源順序才是決定因素。
- **濫用 !important**：一旦多條規則都有 `!important`，又要重新比較優先級，使維護更複雜。

## 重點整理

- 層疊 = 來源 → !important → 優先級 → 來源順序，依序裁決。
- 選擇器優先級：ID (1,0,0) > class (0,1,0) > 標籤 (0,0,1)。
- `!important` 應為最後手段，不是解決衝突的常規工具。

## 自我檢查

1. 一個 `<p>` 同時被 `p { color: red; }` 和 `.text { color: blue; }` 選取，最終顏色是？
2. 為什麼不建議大量使用 `!important`？
````

````md
# margin 與 padding：什麼時候該用哪一個

## 學習目標

- 分辨 `margin` 與 `padding` 在盒模型中的位置差異。
- 用「間距是在元素外部還是內部」來判斷該用哪個。
- 理解背景色與外距合併只影響其中一個。

## 問題情境

很多人知道這兩個都能「加空間」，但不清楚空間加在哪、背景色涵蓋到哪、什麼時候會合併。選錯屬性在視覺上可能看起來一樣，但實際行為差異很大。

## 一句話理解

`padding` 是邊框「以內」的空間（背景色涵蓋）；`margin` 是邊框「以外」的透明空間（背景色不涵蓋）。

## 各自用途

- `padding`：讓元素內容與邊框之間有間隔；背景色會延伸到 padding 區域。
- `margin`：讓元素與周圍其他元素之間有間隔；背景色不延伸到 margin 區域。

## 範例對照

```css
.card {
  padding: 16px;  /* 內容距邊框 16px，背景色涵蓋這個區域 */
  margin: 24px;   /* 卡片與其他元素間隔 24px，透明無背景 */
}
```

## 使用時機判斷

1. 先問：這個空間是「元素內容與邊框之間」的嗎？是 → `padding`。
2. 先問：這個空間是「這個元素與外部元素之間」的嗎？是 → `margin`。

## 常見錯誤

- **用 padding 代替 margin 製造元素間距**：padding 會撐大元素本身，影響背景色範圍和點擊區域。
- **忘記 margin collapse**：相鄰 block 元素的上下 margin 會合併，padding 不會。

## 實務判斷準則與對照表

| 情境 | 建議屬性 |
| --- | --- |
| 文字與按鈕邊框之間的空間 | `padding` |
| 兩個卡片之間的間距 | `margin` |
| 想讓背景色延伸到空間內 | `padding` |
| 水平置中 block 元素 | `margin: 0 auto` |

## 重點整理

- `padding` 在邊框內，`margin` 在邊框外。
- 背景色涵蓋 padding，不涵蓋 margin。
- 上下 margin 會合併，padding 不會。

## 自我檢查

1. 一個按鈕加了 `padding: 12px`，背景色會延伸到 padding 區域嗎？
2. 兩個相鄰 `<p>` 各設 `margin-bottom: 20px` 與 `margin-top: 12px`，實際間距是多少？
````
