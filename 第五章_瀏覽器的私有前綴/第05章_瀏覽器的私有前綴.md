# 第05章 瀏覽器的私有前綴

## 章節導讀

瀏覽器私有前綴主要出現在 CSS 的歷史演進過程中。  
在某些 CSS 新特性尚未成為標準、或不同瀏覽器實作尚未一致時，廠商會先用自己的前綴做實驗性支援。  
現在實務上多半交給建置工具自動補前綴，手動記憶不再是重點。

## 關鍵字

- vendor prefix
- browser prefix
- `-webkit-`
- `-moz-`
- `-ms-`
- `-o-`
- Autoprefixer

## 這一章在解決什麼問題

當你看到同一個 CSS 屬性有多個寫法時，重點不是死背每個前綴，而是要知道：

1. 這些前綴代表什麼
2. 它們通常出現在什麼情境
3. 正式寫法應該放在哪裡
4. 什麼時候可以不再手動寫前綴

## 什麼是瀏覽器私有前綴

瀏覽器私有前綴是瀏覽器廠商為了測試或實驗新功能，先在 CSS 屬性前加上的標記。

它的用途主要是：

- 讓瀏覽器先嘗試支援尚未完全標準化的 CSS 功能
- 區分實驗性寫法與正式標準寫法
- 幫助舊版瀏覽器在標準完成前先提供相容性支援

這一章講的重點，主要是 CSS 的 vendor prefix。  
它不是所有 HTML、CSS、JavaScript 功能都會用到的通用機制。

## 常見前綴

- `-moz-`：Firefox 歷史上常見的前綴
- `-ms-`：IE / 舊版 Microsoft 瀏覽器常見的前綴
- `-webkit-`：Safari、Chrome 等 WebKit / Blink 系瀏覽器常見的前綴
- `-o-`：舊版 Opera 常見的前綴

## 書寫順序

一般寫法是先寫帶前綴的版本，再寫標準寫法。

```css
-webkit-border-radius: 10px;
-moz-border-radius: 10px;
-o-border-radius: 10px;
border-radius: 10px;
```

這樣做的原因是：

- 舊瀏覽器會先吃到它認得的前綴版本
- 支援標準寫法的瀏覽器會以最後的標準屬性為主

## 實務建議

手動記憶所有前綴不是重點。  
現在比較常見的做法是：

- 在需要維護舊專案時才手動補前綴
- 在一般開發中交給建置工具自動處理
- 盡量使用標準寫法作為最終版本

也就是說，前綴是相容性補丁，不是日常開發的核心知識。

## 什麼時候可以去掉前綴

當某個 CSS 屬性已經成為標準，且你的目標瀏覽器都能直接支援未加前綴的寫法時，就可以不再手寫前綴。

如果專案已經有自動加前綴的工具，通常也不需要自己一個個補。

## 常見誤解

- 不是所有 HTML、CSS、JavaScript 功能都需要私有前綴
- 私有前綴主要是 CSS 的歷史用法
- 前綴不是越多越好，能用標準寫法時應以標準寫法為主

## 30 秒複習

如果只記一件事：

> 瀏覽器私有前綴是 CSS 的過渡寫法，先用前綴測試，最後再回到標準屬性。

快速判斷順序：

1. 先寫前綴版本
2. 再寫標準版本
3. 能交給工具就交給工具
4. 維護舊專案才需要特別注意手動補寫
