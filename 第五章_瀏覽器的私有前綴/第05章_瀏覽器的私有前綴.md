# 第05章 瀏覽器的私有前綴

## 章節導讀

瀏覽器私有前綴主要出現在 CSS 的歷史演進過程中。

在某些 CSS 新特性尚未正式成為標準，或不同瀏覽器實作方式尚未完全一致時，瀏覽器廠商會先用自己的前綴來提供實驗性支援。

因此，私有前綴可以理解成：

> CSS 新功能從「實驗階段」走向「正式標準」過程中的相容性寫法。

現在的實務開發中，通常不需要手動記憶所有前綴，而是交給建置工具，例如 Autoprefixer，自動根據專案支援的瀏覽器範圍來補上必要的前綴。

## 這一章在解決什麼問題

當你看到同一個 CSS 屬性有多個寫法時，例如：

```css
-webkit-transform: rotate(45deg);
-moz-transform: rotate(45deg);
-ms-transform: rotate(45deg);
transform: rotate(45deg);
```

重點不是死背每一個前綴，而是要知道：

1. 這些前綴代表什麼
2. 它們通常出現在什麼情境
3. 為什麼標準寫法通常要放在最後
4. 什麼時候需要自己寫前綴
5. 什麼時候應該交給工具處理
6. 什麼時候可以移除前綴

---

## 什麼是瀏覽器私有前綴

瀏覽器私有前綴是瀏覽器廠商為了測試或實驗 CSS 新功能，先在 CSS 屬性或屬性值前面加上的特殊標記。

它的用途主要有三個：

1. 讓瀏覽器先支援尚未完全標準化的 CSS 功能
2. 區分「實驗性寫法」與「正式標準寫法」
3. 讓舊版瀏覽器在標準完成前，也能先提供部分相容性支援

這一章講的重點，主要是 CSS 的 vendor prefix。

它不是所有 HTML、CSS、JavaScript 功能都會用到的通用機制，而是 CSS 發展歷史中常見的相容性手段。

---

## 常見前綴

| 前綴         | 主要對應瀏覽器                   | 說明                      |
| ---------- | ------------------------- | ----------------------- |
| `-webkit-` | Safari、早期 Chrome、部分行動瀏覽器  | WebKit / Blink 系瀏覽器常見前綴 |
| `-moz-`    | Firefox                   | Mozilla Firefox 歷史上常見前綴 |
| `-ms-`     | Internet Explorer、早期 Edge | Microsoft 舊版瀏覽器常見前綴     |
| `-o-`      | 舊版 Opera                  | Opera 舊版 Presto 引擎常見前綴  |

其中，`-webkit-` 在早期行動端與 Safari 相容性中很常見。

---

## 為什麼會有私有前綴

CSS 新功能通常會經過這樣的發展過程：

1. 規格仍在討論中
2. 某些瀏覽器先實作實驗版本
3. 瀏覽器用私有前綴區分實驗寫法
4. 規格逐漸穩定
5. 瀏覽器開始支援標準寫法
6. 舊的前綴寫法逐漸被淘汰

所以私有前綴不是新的語法設計目的，而是標準化過程中的過渡產物。

---

## 書寫順序

一般寫法是：

1. 先寫帶前綴的版本
2. 最後寫標準版本

例如：

```css
-webkit-transform: rotate(45deg);
-moz-transform: rotate(45deg);
-ms-transform: rotate(45deg);
transform: rotate(45deg);
```

這樣做的原因是：

* 舊瀏覽器如果只認得前綴版本，就會使用它能理解的寫法
* 支援標準寫法的瀏覽器，會以後面出現的標準屬性為主
* 標準寫法放最後，可以避免舊的實驗寫法覆蓋正式寫法

簡單記法：

> 前綴寫法在前，標準寫法在後。

---

## 歷史範例：border-radius

早期 `border-radius` 在部分瀏覽器中需要使用前綴：

```css
-webkit-border-radius: 10px;
-moz-border-radius: 10px;
border-radius: 10px;
```

但是現在 `border-radius` 已經是非常成熟的 CSS 屬性，現代專案通常不需要再手動補這些前綴。

這個例子的重點不是叫你現在還要這樣寫，而是幫助理解私有前綴的歷史用法。

---

## 現代實務：通常交給 Autoprefixer

現在實務開發中，通常不會手動記憶每個 CSS 屬性要不要加前綴。

比較常見的做法是：

1. 開發者寫標準 CSS
2. 專案透過 PostCSS / Autoprefixer 處理 CSS
3. Autoprefixer 根據 Browserslist 設定判斷要不要補前綴
4. 最終打包出符合目標瀏覽器需求的 CSS

例如你可能只寫：

```css
.box {
  display: flex;
  user-select: none;
}
```

工具可能會根據目標瀏覽器，自動轉成類似：

```css
.box {
  display: -webkit-box;
  display: -ms-flexbox;
  display: flex;
  -webkit-user-select: none;
  -ms-user-select: none;
  user-select: none;
}
```

不過實際會補哪些內容，取決於你的 Browserslist 設定與目前瀏覽器支援狀況。

---

## Autoprefixer 與 Browserslist 的關係

Autoprefixer 不是隨便把所有前綴都加上去。

它通常會根據 Browserslist 設定來判斷你的專案需要支援哪些瀏覽器版本。

例如：

```txt
> 0.5%
last 2 versions
not dead
```

這代表專案希望支援：

* 全球使用率大於 0.5% 的瀏覽器
* 每個瀏覽器最近兩個版本
* 排除已經停止維護或幾乎沒人使用的瀏覽器

因此，是否需要補前綴，不是看你個人覺得安不安全，而是看專案目標瀏覽器是否需要。

---

## 什麼時候需要手動寫前綴

大部分新專案不太需要手動補前綴。

但以下情況仍然可能會遇到：

1. 維護很舊的專案
2. 專案沒有導入 Autoprefixer
3. 需要支援舊版 IE 或特殊瀏覽器
4. 某些 CSS 寫法在特定瀏覽器仍有相容性差異
5. 你正在閱讀舊程式碼，需要理解為什麼同一個屬性被寫了多次

所以手動前綴比較像是「維護舊系統」或「處理特殊相容性」時才需要關心的知識。

---

## 什麼時候可以去掉前綴

當某個 CSS 屬性已經成為穩定標準，而且你的目標瀏覽器都能直接支援標準寫法時，就可以不再手動寫前綴。

判斷方式通常有三種：

1. 查 Can I use
2. 查看專案的 Browserslist 設定
3. 交給 Autoprefixer 產生結果後再觀察是否仍有前綴

如果專案已經有自動加前綴工具，通常不需要自己一個個刪除或新增。

更好的做法是：

> 讓工具根據目標瀏覽器自動決定，而不是人工猜測。

---

## 實務判斷流程

寫 CSS 時，可以依照下面流程判斷：

1. 先寫標準 CSS
2. 確認專案是否有使用 Autoprefixer
3. 如果有，就交給工具處理
4. 如果沒有，再確認是否真的需要支援舊瀏覽器
5. 如果需要支援舊瀏覽器，再查 Can I use 或參考既有專案寫法
6. 不要為了保險而手動亂加所有前綴

簡化成一句話：

> 新專案先寫標準語法，舊瀏覽器相容性再交給工具或查表處理。

---

## 常見誤解

### 誤解一：所有 CSS 都要加前綴

不是。

只有部分 CSS 屬性在特定歷史階段或特定瀏覽器中需要前綴。

大多數成熟屬性不需要手動加前綴。

---

### 誤解二：前綴越多越安全

不一定。

亂加前綴可能會讓 CSS 變得難讀，也可能保留過時寫法。

正確做法是根據目標瀏覽器需求決定。

---

### 誤解三：看到多個相同屬性就是重複程式碼

不一定。

在舊 CSS 中，多個相似屬性可能是為了支援不同瀏覽器。

例如：

```css
-webkit-user-select: none;
-moz-user-select: none;
-ms-user-select: none;
user-select: none;
```

這不是單純重複，而是相容性處理。

---

### 誤解四：只要有 Autoprefixer 就完全不用懂前綴

也不完全對。

日常開發確實不需要死背，但閱讀舊專案、排查瀏覽器相容性問題時，仍然需要理解前綴的意義。

---

## 閱讀舊專案時要怎麼看

如果你在舊專案中看到很多類似寫法：

```css
-webkit-transition: all 0.3s;
-moz-transition: all 0.3s;
-o-transition: all 0.3s;
transition: all 0.3s;
```

可以這樣判斷：

1. 這通常是為了支援舊瀏覽器
2. 標準寫法通常放在最後
3. 不要急著刪掉前綴
4. 先確認專案需要支援哪些瀏覽器
5. 再決定是否能交給工具或移除舊寫法

在銀行、政府、企業內部系統中，有時候會因為舊瀏覽器或舊設備需求，保留較多相容性寫法。

## 小練習

### 題目一

請問下面寫法中，哪一行通常應該放最後？

```css
-webkit-transform: scale(1.2);
-moz-transform: scale(1.2);
transform: scale(1.2);
```

答案：

```css
transform: scale(1.2);
```

因為它是標準寫法。

---

### 題目二

如果你正在開發一個現代 Vue 專案，是否需要手動記憶所有 CSS 前綴？

答案：

通常不需要。

你應該先確認專案是否有使用 Autoprefixer，並透過 Browserslist 設定目標瀏覽器。

---

### 題目三

如果你在舊專案看到很多 `-webkit-`、`-moz-`、`-ms-`，可以直接刪掉嗎？

答案：

不建議直接刪。

應該先確認：

1. 專案要支援哪些瀏覽器
2. 是否有 Autoprefixer
3. 這些前綴是否仍有相容性需求

## 30 秒複習

如果只記一件事：

> 瀏覽器私有前綴是 CSS 新功能標準化過程中的過渡寫法，現在多半交給工具自動處理。

快速判斷順序：

1. 先理解前綴是相容性寫法
2. 寫 CSS 時優先寫標準語法
3. 前綴版本通常放前面
4. 標準版本通常放最後
5. 現代專案交給 Autoprefixer
6. 是否補前綴，要看 Browserslist 與目標瀏覽器
7. 維護舊專案時，不要隨便刪前綴
