# 從 CSS 筆記產生 Demo 範例

## 使用情境

當需要根據 `notes/<章節>/*.md` 中的 CSS 正式教學筆記，產生 `demos/` 下可直接在瀏覽器開啟的教學 demo 時，套用這份可重複流程。

本檔案負責「正式筆記 → CSS demo」的處理流程，不負責定義角色，也不負責固定輸出格式或單次請求模板。

---

## 必要輸入

- 來源路徑：正式教學筆記所在路徑，應指向 `notes/<章節>/*.md` 或 `notes/<章節>/`。
- 目標輸出路徑：本次 demo 要產生的根目錄，必須位於 `demos/` 下。

## 選填輸入

- 目標知識點或範圍限制：限制本次 demo 產生範圍；若未提供，應根據來源筆記自動分析可做成 demo 的概念。

若任務仍使用 `目標知識點` 欄位，視同 `目標知識點或範圍限制`。

---

## 輸入檢查

1. 檢查來源路徑與目標輸出路徑是否完整。
2. 若必要輸入不足，先指出缺少的項目，不產生 demo。
3. 來源路徑應位於 `notes/` 下；若指向 `origin/`、`appendix/`、`demos/` 或其他目錄，應停止產生，並要求改用正式教學筆記作為來源。
4. 若來源路徑是 `.md` 檔案，應確認它是正式教學筆記，且內容足以支援至少一個 demo 候選。
5. 若來源路徑是章節目錄，應讀取該目錄下與範圍限制相關的 `.md` 檔案；若未提供範圍限制，應讀取該目錄下可形成 demo 的正式筆記。
6. 目標知識點或範圍限制若是整章名稱、筆記主題或大型主題，不應停止產生；應先分析來源筆記，再拆成多個單一概念 demo。
7. 目標知識點或範圍限制若已具體到可做成單一 demo，例如 `justify-content 控制主軸剩餘空間`，則只產生該概念的 demo。
8. 目標輸出路徑必須位於 `demos/` 下，且應是目錄路徑；若指向 `notes/`、`appendix/`、`origin/` 或 `.md` 檔案，應停止產生。
9. 不得修改來源筆記；`notes/` 只作為 demo 的依據。

---

## 來源限制

CSS demo 應根據 `notes/` 中已建立的正式教學內容產生。

- 可以根據正式筆記中的問題情境、範例程式、常見誤解、自我檢查、小結與來源說明設計 demo。
- 可以參考筆記中的 HTML/CSS 範例，另外產生更小、更清楚、更容易觀察的 demo 版本。
- 可以補充必要的 HTML 結構、視覺標籤或操作控制，用來幫助觀察 CSS 結果。
- 不應加入正式筆記沒有支持且與本次範圍或 demo 候選無關的新 CSS 知識。
- 不應把整份筆記改寫成大型綜合 demo。
- `origin/` 可作為追溯來源，但 demo 的直接整理依據仍應是 `notes/`。

若 demo 必須補充正式筆記未明說的 CSS 背景，應標示為「補充說明」，並保持篇幅簡短。

---

## 來源分析流程

正式產生 demo 前，先完成來源分析。此分析可以作為內部工作步驟，不一定要完整輸出，除非任務明確要求。

### 1. 讀取與建立內容地圖

讀取來源後，先建立筆記內容地圖：

- 最近的標題與段落。
- 學習目標與本章範圍中相關的描述。
- 直接支持 CSS 概念的 HTML、CSS 或表格。
- 筆記中的問題情境、畫面觀察、最小修改與結果解釋。
- 常見誤解、失效原因與檢查重點。
- 自我檢查題中可轉成 demo 觀察任務的題目。

若提供目標知識點或範圍限制，應用它過濾內容地圖；若未提供，應以整份來源筆記作為分析範圍。

### 2. 建立 Demo 候選清單

根據內容地圖建立 demo 候選清單。每個候選 demo 應包含：

- Demo 名稱。
- 主要 CSS 概念。
- 來源標題或來源段落。
- 需要的最小 HTML 結構。
- 主要 CSS 宣告與必要前置宣告。
- 建議 demo 型態。
- 學習者應該在畫面中看到的變化。
- 最容易發生的誤解或失效原因。
- 建議輸出子目錄名稱。

建立候選清單時應遵守：

- 若範圍限制是整章或大型主題，拆成多個候選 demo。
- 若範圍限制已經是單一概念，只保留與該概念直接相關的候選 demo。
- 若一個候選包含多個互相獨立的觀察任務，應拆成多個候選 demo，不要放進同一個範例。
- 若某個概念缺少足夠來源支持，應略過或標示為不產生。

候選清單可以作為內部工作步驟；若任務要求說明產生策略，才需要輸出完整候選清單。

### 3. 選擇 Demo 型態

依每個 demo 候選選擇最適合的 demo 型態：

| Demo 型態 | 適用情境 |
|---|---|
| 單一狀態 | 展示一個屬性或值的基本效果。 |
| 修改前 / 修改後 | 需要看出加入某條 CSS 宣告前後的差異。 |
| 錯誤 / 正確 | 主題重點是常見誤解、作用對象錯誤或失效原因。 |
| 少量互動切換 | 需要比較少數屬性值或 class 狀態，且互動能幫助觀察。 |

預設優先使用靜態對照。只有互動能明顯提升理解時，才加入 JavaScript。

## Demo 產生流程

1. 依輸入讀取來源筆記。
2. 建立來源筆記內容地圖。
3. 根據目標知識點或範圍限制建立 demo 候選清單；若未提供範圍限制，從整份筆記建立候選清單。
4. 刪除缺少來源支持、範圍外或不適合做成可觀察範例的候選。
5. 為每個保留候選決定主要觀察問題、demo 型態與輸出子目錄。
6. 逐一設計最小 HTML 結構，讓 CSS 的作用對象清楚可見。
7. 逐一設計最小 CSS 規則，讓主要宣告造成的畫面變化清楚可觀察。
8. 依需要建立靜態對照或少量互動；若某個 demo 不需要互動，不建立該 demo 的 `main.js`。
9. 為每個 demo 產生檔案，預設包含 `index.html`、`style.css`、必要時的 `main.js` 與 `README.md`。
10. 在每個 README 中說明來源筆記、對應知識點、開啟方式、觀察重點與常見誤解。
11. 檢查每個 demo 是否符合 `prompts/system/css-demo-role.md` 的可執行性、粒度與目錄規則。
12. 將完成內容寫入目標輸出路徑下的對應子目錄。

## 檔案與目錄

目標輸出路徑是本次任務的 demo 輸出根目錄，必須位於：

```text
demos/
```

建議結構：

```text
demos/
  <章節>/
    <教學筆記>/
      <demo-name>/
        index.html
        style.css
        main.js
        README.md
```

若本次產生多個 demo，應在目標輸出路徑下依來源筆記與 demo 名稱建立子目錄，不要把多個 demo 檔案混放在同一層。

若本次只產生單一小型 demo，也可以使用：

```text
demos/
  <章節>/
    <教學筆記>/
      index.html
      style.css
      README.md
```

檔案使用原則：

- `index.html` 必須存在，並能直接在瀏覽器開啟。
- `style.css` 用於放主要 CSS 規則；若使用單檔 demo，可將 CSS 放在 `index.html` 的 `<style>` 中。
- `main.js` 只在需要互動或狀態切換時建立。
- `README.md` 用於說明 demo 目的與觀察方式；若使用單一 `index.html`，仍應在頁面或檔案中保留足夠說明。
- 多個 demo 共用同一來源筆記時，子目錄名稱應反映各自的主要概念，例如 `justify-content-main-axis` 或 `align-content-multiple-lines`。

## 產出前品質檢查

### 輸入與來源

- 來源路徑是否位於 `notes/`。
- 目標知識點或範圍限制是否已正確套用；若未提供，是否已根據來源筆記建立 demo 候選清單。
- 目標輸出路徑是否位於 `demos/`。
- 每個 demo 是否只根據正式筆記中可支持的內容設計。
- 是否沒有修改 `notes/` 中的正式教學筆記。

### Demo 品質

- 是否能直接用瀏覽器開啟。
- 是否只聚焦一個主要知識點。
- 是否能清楚觀察主要 CSS 宣告造成的畫面變化。
- 是否清楚呈現 CSS 的作用對象，例如父元素、子元素、文字或盒子。
- 是否避免把父項屬性寫到子元素上，或把子項屬性寫到父元素上。
- 是否避免把無關 CSS 技巧、框架、外部 CDN 或建置工具放進 demo。
- 是否沒有引用不存在的 CSS、JS、圖片或其他資產。
- 是否沒有把正式筆記大段複製進 demo 頁面。
- 若有互動，互動是否只服務於觀察 CSS 行為。

### 文件品質

- README 是否標明來源筆記與對應知識點。
- README 是否說明開啟方式、操作方式與觀察重點。
- README 是否讓學習者知道要觀察什麼，而不是只列檔案清單。
- README 是否指出對應的常見誤解或檢查重點。
- README 是否避免重新撰寫整份正式筆記。

---

## 輸出

完成後，將 demo 檔案寫入本次指定的目標輸出路徑。

若環境無法實際寫入檔案，應輸出完整檔案內容，並明確標示建議寫入路徑。
