# 瀏覽器的私有前綴

## 學習目標

讀完本章後，你應該能說明瀏覽器私有前綴為什麼會出現在 CSS 屬性名稱前面，以及它和「新功能測試」、「標準化過程」與「舊版瀏覽器相容」之間的關係。

你也應該能判斷一組同功能 CSS 宣告的基本書寫順序：先放瀏覽器私有前綴寫法，再放標準寫法。當看到 `-webkit-`、`-moz-`、`-ms-` 或 `-o-` 這類寫法時，你應該知道它們不是新的 CSS 類型，而是瀏覽器廠商為特定功能準備的前綴版本。

本章的實務重點不是背誦所有前綴，而是建立一個判斷：平常不要把私有前綴當成主要開發負擔；只有在需要支援特定舊版瀏覽器或實驗性功能時，才需要確認是否要保留或補上前綴。

## 前置知識

學習本章前，建議先能看懂基本 CSS 規則，例如：

```css
.box {
    border-radius: 10px;
}
```

你需要知道 `.box` 是 selector，`border-radius` 是 property，`10px` 是 value。瀏覽器讀取 CSS 時，會嘗試理解每一條 declaration，並把能理解的樣式套用到符合 selector 的元素上。

本章會提到 CSS3 新特性、瀏覽器支援、標準屬性與建構工具。這些內容只作為理解私有前綴的背景，不會展開成完整的 CSS 規格流程、瀏覽器版本支援表或前端工程化章節。

## 本章範圍

本章深入處理的是 CSS 屬性名稱前面的瀏覽器私有前綴，例如 `-webkit-`、`-moz-`、`-ms-` 與 `-o-`。主線是：為什麼某些 CSS 特性曾經需要加前綴、加前綴時應該如何排序、以及什麼時候可以不用手動關注它。

來源中提到瀏覽器私有前綴也可能出現在 HTML、CSS 和 JavaScript 等標準規範實作中。本章只聚焦 CSS property 的教學，不展開 JavaScript API 前綴、HTML 屬性差異、完整瀏覽器內核史或每個 CSS 屬性的版本支援。

來源也提到現代常用 CSS3 新特性在主流瀏覽器中通常已有支援，且可以借助現代建構工具補前綴。本章會保留這個實務方向，但不列出具體瀏覽器版本，因為版本支援會變動，正式專案仍應以目標瀏覽器需求為準。

## 先看問題：同一個圓角效果，為什麼會出現好幾條相似宣告

初學者看到下面這段 CSS 時，常會覺得很奇怪：明明只是要讓盒子變成圓角，為什麼要寫四行？

```css
.card {
    -moz-border-radius: 10px;
    -webkit-border-radius: 10px;
    -o-border-radius: 10px;
    border-radius: 10px;
}
```

這四行看起來都在描述圓角，但前三行的 property 前面多了不同前綴。這些前綴不是 selector，也不是 value，而是 property name 的一部分。它們表示瀏覽器廠商曾經用自己的方式提供某個尚未完全穩定或尚未普遍支援的 CSS 功能。

學習本章時，可以先建立一個畫面模型：標準 CSS 屬性是正式路線，私有前綴屬性是某些瀏覽器曾經提供的臨時路線。當瀏覽器已經支援標準寫法時，開發者就應該讓標準寫法成為主要結果。

補充說明：這裡使用 `border-radius` 是依照來源範例說明書寫順序，不代表現代專案一定還需要替 `border-radius` 手動加前綴。來源也提醒，較新版本瀏覽器通常不需要額外添加前綴。

## 第一課：為什麼新 CSS 功能曾經需要私有前綴

### 問題情境

假設某個瀏覽器想先讓開發者試用一個新的 CSS 功能，但這個功能還沒有被所有瀏覽器穩定支援，語法或行為也可能還會調整。如果瀏覽器直接使用未加前綴的正式屬性名稱，開發者可能會誤以為這個功能已經完全標準化。

私有前綴就是用來處理這種過渡期的標記。

### 畫面觀察

來源說明：W3C 標準提出某個 CSS 特性後，在瀏覽器正式支援之前，瀏覽器廠商可能會根據自己的瀏覽器內核，用私有前綴測試該 CSS 特性。等瀏覽器正式支援後，就不需要私有前綴。

因此，看到下面這種寫法時：

```css
.box {
    -webkit-border-radius: 10px;
    border-radius: 10px;
}
```

可以把它理解成兩個層次：

第一行是特定瀏覽器曾經可能理解的前綴版本。第二行是標準版本。它們的目標通常是讓同一種視覺功能在不同支援條件下盡量可以生效。

### 最小修改

如果一開始只寫標準屬性：

```css
.box {
    border-radius: 10px;
}
```

對於已經支援標準寫法的瀏覽器，這樣就足夠了。若某些舊版瀏覽器只認得前綴版本，來源建議可以補上私有前綴寫法：

```css
.box {
    -webkit-border-radius: 10px;
    border-radius: 10px;
}
```

這裡的重點不是把 `border-radius` 當成現在一定要加前綴的屬性，而是看懂「前綴版本」和「標準版本」的關係。

### 結果解釋

加入前綴後，支援特定前綴寫法的瀏覽器有機會讀到它能理解的 declaration；支援標準寫法的瀏覽器則可以使用標準屬性。這讓同一份 CSS 在不同瀏覽器支援程度下有較高機會呈現接近的結果。

補充說明：CSS 的容錯特性會讓瀏覽器忽略自己不理解的 declaration。也就是說，某個瀏覽器如果不認得 `-webkit-border-radius`，通常會跳過這條宣告，繼續讀下一條。這也是多寫幾條前綴版本能形成相容寫法的基礎。

### 誤判修正

不要把私有前綴理解成「比標準屬性更正式」或「更高級」。它反而通常代表這個功能曾經處在實驗、測試或相容過渡階段。

也不要把「曾經需要前綴」推論成「永遠都要手寫前綴」。來源明確指出，在瀏覽器正式支援該 CSS 特性後，就不需要私有前綴。現代開發中，常用 CSS 功能通常應優先使用標準寫法。

### 小整理

私有前綴的核心用途，是讓瀏覽器廠商在標準功能完全穩定或普遍支援前，先提供一種帶有廠商標記的測試或相容寫法。學習時要抓住「過渡期」這個概念，而不是把前綴當成日常必背清單。

## 第二課：加前綴時，為什麼標準寫法要放在最後

### 問題情境

如果你確定需要同時保留前綴版本和標準版本，下一個問題是：哪一行應該先寫？來源建議的順序是先寫私有前綴屬性，再寫標準屬性。

這個順序可以讓 CSS 從舊的相容寫法走向新的標準寫法。

### 畫面觀察

來源提供的提倡寫法如下：

```css
.box {
    -moz-border-radius: 10px;
    -webkit-border-radius: 10px;
    -o-border-radius: 10px;
    border-radius: 10px;
}
```

這段 CSS 的教學重點是順序：前綴版本放前面，標準版本 `border-radius` 放最後。

修改前，如果只看未加前綴的標準寫法：

```css
.box {
    border-radius: 10px;
}
```

支援標準屬性的瀏覽器會讓 `.box` 顯示圓角。若某些舊版瀏覽器只理解前綴寫法，可能就看不到預期效果。

### 最小修改

在需要支援前綴版本時，可以改成：

```css
.box {
    -moz-border-radius: 10px;
    -webkit-border-radius: 10px;
    -o-border-radius: 10px;
    border-radius: 10px;
}
```

HTML 可以保持很小：

```html
<div class="box">圓角盒子</div>
```

為了讓視覺結果更容易觀察，可以搭配最少的尺寸與背景：

```css
.box {
    width: 160px;
    padding: 20px;
    background: #e8f3ff;

    -moz-border-radius: 10px;
    -webkit-border-radius: 10px;
    -o-border-radius: 10px;
    border-radius: 10px;
}
```

### 結果解釋

`.box` 是 CSS 的作用對象。`width`、`padding` 和 `background` 只是讓盒子可見；本課的主角是圓角相關宣告。

修改前，只有標準 `border-radius`。在支援標準寫法的瀏覽器中，盒子四角會變圓。修改後，CSS 多了幾條帶前綴的候選寫法，讓需要前綴的舊環境也有機會套用類似效果。

標準屬性放在最後，是為了在瀏覽器同時理解前綴寫法與標準寫法時，讓標準寫法成為最後的主要宣告。這個順序也表達維護意圖：前面的前綴是相容補充，最後的標準寫法才是希望長期保留的主線。

### 誤判修正

不要把這種寫法誤解成「每個 CSS 屬性都要寫四次」。只有在某個 CSS 功能確實需要支援特定前綴版本時，才有必要這樣處理。

也不要把來源中的 `border-radius` 範例當成現代專案的固定模板。它適合用來理解「私有前綴在前、標準屬性在後」的順序，但是否還需要前綴，必須回到目標瀏覽器支援條件判斷。

### 小整理

需要保留前綴時，書寫順序可以先放私有前綴版本，再放標準版本。這樣既能照顧舊支援條件，也能讓標準 CSS 寫法保持在最重要的位置。

## 第三課：看到 `-webkit-`、`-moz-`、`-ms-`、`-o-` 時，應該先判斷什麼

### 問題情境

你在舊專案中看到很多帶有前綴的 CSS：

```css
.panel {
    -webkit-transform: scale(1);
    -ms-transform: scale(1);
    transform: scale(1);
}
```

初學者容易卡在兩個方向：要不要把每個前綴背起來？這些前綴是不是代表不同 CSS 功能？

比較好的讀法是先把前綴當作瀏覽器廠商標記，再回到後面的屬性名稱看它想處理哪個 CSS 功能。

### 畫面觀察

來源列出常見私有前綴：

| 前綴 | 來源中的對應瀏覽器 | 觀察重點 |
|---|---|---|
| `-moz-` | Firefox | 表示 Firefox 相關的私有屬性寫法 |
| `-ms-` | IE | 表示 Internet Explorer 相關的私有屬性寫法 |
| `-webkit-` | Safari、Chrome | 表示 Safari、Chrome 相關的私有屬性寫法 |
| `-o-` | Opera | 表示 Opera 相關的私有屬性寫法 |

這些前綴通常放在 property name 前面，而且包含連字號。例如 `-webkit-border-radius` 是帶有 `-webkit-` 前綴的 property name，不是 `border-radius` 的 value。

### 最小修改

先看沒有前綴的標準寫法：

```css
.panel {
    transform: scale(1);
}
```

如果需要保留特定舊支援條件，可能會看到：

```css
.panel {
    -webkit-transform: scale(1);
    -ms-transform: scale(1);
    transform: scale(1);
}
```

這裡的核心功能仍然是 `transform`。`-webkit-` 和 `-ms-` 只是表示某些瀏覽器曾經可能需要不同 property name 才能理解這個功能。

### 結果解釋

`.panel` 是作用對象。修改前，只有標準 `transform`。修改後，CSS 先提供前綴版本，再提供標準版本。

對學習者來說，最重要的觀察不是「這裡有幾個前綴」，而是「這些宣告是否在描述同一個視覺功能」。如果它們都設定類似的值，就可以判斷這是一組為相容性準備的寫法。

### 誤判修正

不要看到 `-webkit-` 就以為這份 CSS 只能在 Safari 或 Chrome 使用。前綴表示某個瀏覽器家族曾經提供的私有寫法，但實際支援狀況會受版本與實作影響。正式專案應以目標瀏覽器測試或支援資料為準。

也不要把前綴清單當成完整瀏覽器歷史。來源提供的是入門對應關係，本章只用它幫助你閱讀 CSS，不展開瀏覽器內核演變。

### 小整理

讀到私有前綴時，先分成兩部分看：前綴指出廠商或瀏覽器家族，後面的 property name 才指出真正想控制的 CSS 功能。學習目標是能讀懂它的相容意圖，不是背出所有曾經存在的前綴屬性。

## 第四課：什麼時候可以不用手動處理私有前綴

### 問題情境

學完前綴後，初學者可能會反過來擔心：是不是每次寫 CSS 都要查一遍所有瀏覽器前綴？來源一開始就提醒，不需要把私有前綴當成日常主要負擔。

原因是常用 CSS3 新特性在主流瀏覽器中通常已有支援；即使為了舊瀏覽器需要補前綴，也可以借助現代建構工具處理。

### 畫面觀察

把手寫前綴當成日常習慣，會帶來兩個問題。

第一，CSS 會變得冗長。每一個功能都手動複製多個前綴版本，會讓真正重要的標準屬性被淹沒。

第二，前綴需求會變動。某個屬性是否需要前綴，取決於目標瀏覽器、版本與功能支援狀態。若只靠記憶，很容易留下已經不需要的舊寫法，或漏掉真正需要處理的相容條件。

### 最小修改

日常撰寫時，可以先以標準寫法作為主線：

```css
.button {
    border-radius: 8px;
    transition: background-color .2s;
}
```

如果專案有支援舊版瀏覽器的需求，再依照目標環境決定是否產生或保留前綴版本：

```css
.button {
    -webkit-transition: background-color .2s;
    transition: background-color .2s;
}
```

補充說明：實務上常見做法是讓 CSS 後處理或建構流程根據專案設定的目標瀏覽器，自動判斷是否需要補上前綴。這能降低人工記憶成本，也能讓前綴需求跟著專案支援範圍調整。

### 結果解釋

標準寫法讓 CSS 保持清楚，前綴寫法只在相容需求成立時出現。這符合來源的方向：不用刻意記憶每個前綴，也不用每個屬性都查一遍。

若一個屬性已經成為標準，並且被目標範圍內的主流瀏覽器普遍支援，就可以考慮移除前綴。來源的說法是：當一個屬性成為標準，並且被 Firefox、Chrome 等主流瀏覽器最新版普遍相容時，可以去掉該屬性的瀏覽器前綴。

### 誤判修正

不要把「不用關注前綴」理解成「永遠不需要相容性判斷」。如果專案明確要求支援舊版瀏覽器，或正在使用實驗性 CSS 功能，就仍然需要檢查目標環境。

也不要把「主流瀏覽器支援」寫成固定版本結論。來源沒有提供版本表，本章也不做具體版本承諾。實作時應以專案目標瀏覽器與當下支援資料為準。

### 小整理

私有前綴在現代 CSS 學習中應該被理解為相容性背景，而不是每天手寫的主要任務。日常先寫標準 CSS；有舊版瀏覽器或實驗功能需求時，再用工具或支援資料確認是否要補前綴。

## 私有前綴的判斷流程

學完前面幾課後，可以用下面的流程判斷一段 CSS 中的私有前綴：

1. 先看標準屬性是什麼，例如 `border-radius`、`transform` 或 `transition`。
2. 再看前綴版本是否只是為了支援同一個功能的舊寫法。
3. 如果要保留前綴，將前綴版本放在標準版本前面。
4. 如果專案只支援已普遍支援標準寫法的現代瀏覽器，可以考慮只保留標準寫法。
5. 如果專案有明確舊版瀏覽器需求，不要靠記憶決定，應依目標瀏覽器支援條件或建構工具輸出判斷。

這個流程的核心是「標準寫法優先理解，相容寫法按需求保留」。私有前綴不是越多越好，也不是看到就一定要刪掉；它要服務於具體支援範圍。

## 常見誤解與修正方式

### 誤解一：私有前綴是 CSS 的必背語法清單

私有前綴確實是 CSS 歷史與相容性的一部分，但來源一開始就提醒，不需要絞盡腦汁記憶，也不需要每個都查。常用功能通常優先寫標準屬性；需要相容舊環境時，再依專案條件處理。

修正方式是先理解前綴的目的：它是實驗、測試或舊版相容的標記，不是日常撰寫 CSS 的主線。

### 誤解二：加了所有前綴就一定比較好

前綴不是裝飾，也不是越多越安全。無需求地保留大量舊前綴，會讓 CSS 難讀，也可能讓學習者誤判哪些寫法仍然必要。

修正方式是根據目標瀏覽器支援範圍決定。若標準屬性已能覆蓋專案需求，就不必為了看起來完整而保留舊前綴。

### 誤解三：標準屬性可以省略，只寫前綴版本

來源建議的順序是先寫私有前綴，再寫標準屬性。若只寫前綴版本，支援標準寫法的瀏覽器雖然可能仍有相容行為，但 CSS 的主要意圖會變得不清楚，也不利於長期維護。

修正方式是讓標準屬性成為最後且主要的寫法。前綴版本只作為必要的相容補充。

### 誤解四：來源範例中的 `border-radius` 代表現在一定要加前綴

來源使用 `border-radius` 示範前綴書寫順序，不代表每個現代專案都需要這樣寫。來源也說明較新版本瀏覽器通常不需要添加前綴。

修正方式是把範例的學習重點放在順序，而不是把它當成固定相容模板。

### 誤解五：看到前綴就可以直接刪掉

舊專案中的前綴可能是為了支援特定使用者環境。若沒有先確認專案目標瀏覽器，直接刪除可能造成舊環境中的樣式退化。

修正方式是先確認支援範圍。若確定目標環境都支援標準屬性，再移除前綴會比較穩妥。

## 實務使用情境

私有前綴最常出現在舊專案、歷史 CSS 範例、需要支援舊版瀏覽器的商業專案，以及曾經屬於實驗階段的 CSS 功能中。閱讀這些程式碼時，應該先找出它們對應的標準屬性，再判斷前綴是否仍然服務於目前專案的支援範圍。

在新專案中，通常先寫標準 CSS，讓程式碼保持清楚。如果專案使用建構流程，前綴可以交由工具依目標瀏覽器設定處理。這樣比手動背誦每個前綴更穩定，也更符合來源提到的現代開發方向。

在維護舊 CSS 時，私有前綴可以當成相容性線索。它告訴你：這段樣式可能曾經為某些瀏覽器做過相容處理。是否保留，取決於該專案現在還要不要支援那些環境。

## 小結

瀏覽器私有前綴是 CSS 功能從實驗、測試或舊版相容走向標準支援時留下的寫法。它通常出現在 property name 前面，例如 `-webkit-`、`-moz-`、`-ms-` 與 `-o-`，用來標示特定瀏覽器廠商或瀏覽器家族曾經提供的私有實作。

學習本章時要抓住三個判斷：第一，標準屬性才是長期主線；第二，需要前綴時，先寫前綴版本，再寫標準版本；第三，現代開發不應靠背誦手動維護所有前綴，而應根據目標瀏覽器與建構工具判斷是否需要保留。

## 來源與補充說明

本章主要根據 `origin/第05章_瀏覽器的私有前綴/瀏覽器的私有前綴.md` 整理。

來源直接支持的內容包括：瀏覽器私有前綴用於瀏覽器正式支援某 CSS 特性之前的測試或實驗；私有前綴主要用來區分實驗性功能與標準功能；常見前綴包含 `-moz-`、`-ms-`、`-webkit-` 與 `-o-`；較新版本瀏覽器通常無須添加前綴；需要書寫時應先寫私有前綴屬性，再寫標準屬性；當屬性成為標準並被主流瀏覽器最新版普遍相容時，可以去掉前綴；現代建構工具可以協助添加私有前綴。

背景內容：來源提到私有前綴也可能涉及 HTML、CSS 和 JavaScript 等標準規範實作。本章只保留這個背景，不展開 CSS 以外的前綴行為。

補充說明：本章補充了「瀏覽器通常會忽略自己不理解的 CSS declaration」、「前綴版本與標準版本可視為同一功能在不同支援條件下的候選寫法」、「日常先寫標準 CSS，再依目標瀏覽器或建構流程補前綴」等教學說明。這些補充用來幫助初學者理解來源內容，未列舉具體瀏覽器版本支援。

可另開章節：CSS 規格標準化流程、瀏覽器支援資料查詢、前端建構工具與自動補前綴流程、完整 CSS 相容性測試、JavaScript 或 HTML 中的廠商前綴。

不應納入本章：完整瀏覽器內核歷史、所有曾經需要前綴的 CSS 屬性清單、各瀏覽器版本支援表。這些內容會讓本章從「私有前綴概念」發散成「瀏覽器相容性資料庫」。

待確認：來源未提供具體 CSS 屬性的當前版本支援狀態，也未提供建構工具設定方式。因此本章不判定任何特定屬性在 2026 年是否仍需前綴，也不指定某個工具的完整使用流程。
