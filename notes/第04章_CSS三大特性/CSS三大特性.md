# CSS 三大特性：層疊性、繼承性、優先級

## 學習目標

讀完本章後，你應該能判斷同一個 HTML 元素最後會套用哪一個 CSS 樣式。當多條規則同時影響同一個元素時，你應該能分辨這是層疊性、繼承性還是優先級造成的結果。

你也應該能看懂 specificity（權重）的基本計算方式，知道複合選擇器的權重會疊加但不會進位，並能說明為什麼「父元素權重很高」不代表子元素一定會使用父元素的樣式。

## 前置知識

學習本章前，建議先知道 HTML 的父子結構，例如 `<div>` 裡面可以包含 `<p>`、`<a>` 或 `<h1>`。你也需要能看懂基本 CSS 規則，例如：

```css
p {
    color: red;
}
```

本章會出現標籤選擇器、類選擇器、ID 選擇器與行內樣式。你不需要先完整掌握所有選擇器種類，但要能理解：selector 決定 CSS 作用在哪些元素上，declaration 決定要修改哪些樣式。

## 本章範圍

本章深入處理 CSS 三大特性：層疊性、繼承性與優先級。主線是「瀏覽器如何替同一個元素、同一個 CSS 屬性決定最後使用哪個值」。

來源中關於字體屬性、文字屬性、`color` 與 `line-height` 的繼承會納入本章，因為它們能幫助理解繼承性。來源中的 `a` 標籤與 `h1` 標籤案例也會保留，但會用「瀏覽器預設樣式直接套用到元素」來修正「繼承失效」這個容易誤解的說法。

本章只會簡短提到 `!important`，因為來源將它放在優先級公式中。完整的 cascade origin、cascade layer、瀏覽器開發者工具、所有 selector 類型與 `!important` 的實務管理策略，適合另開章節。

## 先看問題：為什麼同一段文字最後不是你以為的顏色

初學 CSS 時，常見困惑是：明明已經寫了 `color: red;`，畫面上的文字卻不是紅色。這不一定是 CSS 沒有生效，而可能是同一個元素同時被多條規則影響。

例如下面這個元素：

```html
<div class="message" id="notice">CSS 三大特性</div>
```

它可能同時被 `div`、`.message`、`#notice`，甚至行內樣式選到。若它放在某個父元素裡，也可能從父元素繼承文字顏色或字體大小。

因此，學習本章時要先建立一個心智模型：瀏覽器不是只看「你有沒有寫某條 CSS」，而是會替每個元素、每個 CSS 屬性收集可能的值，再按照規則決定最後採用哪一個。

可以先把三大特性理解成四個問題：

1. 這個元素自己有沒有被 CSS 直接指定這個屬性？
2. 如果有多條直接指定互相衝突，哪一條權重比較高？
3. 如果權重相同，哪一條在層疊順序中比較後面？
4. 如果元素自己沒有指定，這個屬性能不能從父元素繼承？

接下來每一課都用一個小問題觀察這套判斷方式。

## 第一課：同一個選擇器重複設定時，後面的規則為什麼會勝出

### 問題情境

你想讓頁面上的 `<div>` 文字變成紅色，後來又在 CSS 後面補了一段新的 `div` 規則，把顏色改成粉紅色。這時畫面最後會顯示哪一個顏色？

### 畫面觀察

HTML 只有一個 `<div>`：

```html
<div>長江後浪推前浪</div>
```

CSS 中同一個 selector 出現兩次：

```css
div {
    color: red;
    font-size: 12px;
}

div {
    color: pink;
}
```

兩條規則都選到同一個 `<div>`，而且都設定了 `color`。這就形成樣式衝突：同一個元素的同一個屬性，同時有兩個可能的值。

### 最小修改

如果只觀察衝突屬性，可以把範例縮到最小：

```css
div {
    color: red;
}

div {
    color: pink;
}
```

### 結果解釋

最後畫面中的 `<div>` 會顯示粉紅色。原因是兩條規則的 selector 完全相同，權重也相同；當權重相同且設定的是同一個屬性時，後寫的規則會覆蓋前面衝突的規則。

這就是來源中所說的層疊性。層疊不是「後面的整段規則把前面的整段規則清掉」，而是針對同一個元素、同一個屬性的衝突做取捨。在原本的完整範例中，`font-size: 12px;` 沒有被第二段規則重新設定，所以仍然會保留。

### 誤判修正

不要把「就近原則」理解成「離 HTML 標籤實體距離最近」。在這個範例中，比較的是 CSS 來源順序：同權重、同屬性衝突時，後出現的 declaration 會勝出。

也不要以為後面的 selector 一出現，前面同 selector 的所有樣式都失效。只有互相衝突的同一個屬性需要決定勝負；沒有衝突的屬性仍會一起作用。

### 小整理

層疊性主要解決樣式衝突問題。當多條 CSS 規則選到同一個元素，而且設定同一個屬性時，才需要判斷哪個值勝出。若 selector 權重相同，通常後寫的規則會覆蓋前面衝突的規則。

## 第二課：子元素沒有自己設定樣式時，為什麼會跟著父元素變化

### 問題情境

你希望整個頁面的文字都使用同一種基礎字體、字級與行高，不想在每個 `<p>`、`<li>`、`div` 上重複設定。這時可以利用繼承性，讓子元素從父元素取得某些樣式。

### 畫面觀察

先看一段 HTML：

```html
<div>
    <p>龍生龍，鳳生鳳</p>
</div>
```

如果 CSS 只寫在 `div` 上：

```css
div {
    color: pink;
    font-size: 20px;
}
```

`<p>` 自己沒有設定 `color` 和 `font-size`，但畫面上 `<p>` 裡的文字仍然會呈現粉紅色與 `20px`。原因是這些文字相關屬性通常可以從父元素繼承。

### 最小修改

如果你想設定整頁基礎文字樣式，常見做法是把基礎字體設定在 `body`：

```css
body {
    font: 16px/1.5 "Microsoft YaHei", sans-serif;
}
```

這段 CSS 的意思是：`body` 的文字大小是 `16px`，行高是目前字體大小的 `1.5` 倍，並指定字體家族。

### 結果解釋

繼承性的重點是：子元素如果沒有自己設定某個可繼承屬性，就會沿著父元素、祖先元素往上尋找可以繼承的值。

來源提供了一個實用判斷：常見會繼承的屬性多半和文字呈現有關，例如 `color`、`font-*`、部分 `text-*` 與 `line-height`。常見不會繼承的屬性多半和盒模型或版面有關，例如 `border`、`background`、`padding`、`margin`、`width`、`height` 與 `overflow`。

這個規律能幫助初學者建立直覺：文字樣式常常需要整個區塊保持一致，所以適合繼承；盒子的邊框、背景、內距、外距與尺寸如果全部自動傳給子元素，版面很容易失控，因此通常不會繼承。

### 行高的繼承

來源特別提到 `line-height` 可以不帶單位。這在繼承時很重要：

```css
body {
    font: 12px/1.5 "Microsoft YaHei", sans-serif;
}

div {
    font-size: 14px;
}

p {
    font-size: 16px;
}
```

在這個範例中，`body` 設定的 `line-height` 是 `1.5`。`div` 繼承到的是這個比例，所以 `div` 的行高會以自己的 `14px` 字級計算，結果是 `21px`。`p` 的字級是 `16px`，所以行高會是 `24px`。

補充說明：無單位 `line-height` 繼承時保留比例，子元素會用自己的 `font-size` 重新計算。這也是為什麼基礎文字設定中常見 `font: 16px/1.5 ...` 這類寫法。

### 誤判修正

不要以為所有 CSS 屬性都會繼承。父元素設定 `color`，子元素文字通常會跟著變；父元素設定 `border`，子元素不會自動得到相同邊框。

也不要以為繼承只看最外層祖先。若多層祖先都設定了同一個可繼承屬性，子元素會優先繼承離自己較近的祖先值。

### 小整理

繼承性可以減少重複 CSS，特別適合基礎文字樣式。判斷時先問：子元素自己有沒有設定這個屬性？這個屬性通常會不會繼承？最近的祖先元素是否提供了可繼承的值？

## 第三課：看起來沒有繼承時，真的代表繼承失效嗎

### 問題情境

你在父元素上設定文字顏色和字級，卻發現 `<a>` 還是看起來像超連結的顏色，`<h1>` 也沒有變成你設定的小字。來源把這種情況稱為「繼承失效的特殊情況」。

### 畫面觀察

先看範例：

```html
<div>
    <a href="#">超連結</a>
    <h1>一級標題</h1>
</div>
```

```css
div {
    color: red;
    font-size: 12px;
}
```

直覺上，`<a>` 和 `<h1>` 都在 `<div>` 裡，應該能繼承 `color` 和 `font-size`。但實際畫面常常不是這樣：超連結可能仍然呈現瀏覽器預設的連結顏色，`h1` 也可能仍然比一般文字大。

### 最小修改

若你真的希望它們使用指定樣式，可以直接選到這些元素：

```css
div a {
    color: red;
}

div h1 {
    font-size: 12px;
}
```

### 結果解釋

這裡更精確的理解不是「繼承機制壞掉」，而是：瀏覽器本身可能已經用預設樣式直接選到了 `<a>` 或 `<h1>`。當元素自己被某條規則直接指定樣式時，直接指定通常會比從父元素繼承來的值更優先。

也就是說，`<a>` 不是不能使用紅色，而是它可能已經有直接作用在自己身上的連結顏色規則。`<h1>` 也不是不能繼承字級，而是它通常有瀏覽器預設的標題字級規則。

### 誤判修正

不要把所有「沒有繼承到」都當成繼承失效。除錯時要多問一步：這個元素自己是不是已經被其他 CSS 或瀏覽器預設樣式直接指定了同一個屬性？

來源中用 `a` 的 `color` 和 `h` 系列標籤的 `font-size` 作為特殊情況，適合初學記憶。但正式判斷時，應把它放回本章主線：直接作用在元素上的樣式，會比繼承值更有競爭力。

### 小整理

繼承值的權重可以先理解為 0。只要子元素自己被直接選中並設定了同一個屬性，即使那條直接規則看起來很簡單，也可能覆蓋從父元素繼承來的結果。

## 第四課：不同選擇器同時選到同一元素時，誰的優先級比較高

### 問題情境

現在換一種情況：不是同一個 selector 重複出現，而是不同 selector 同時選到同一個元素。這時不能只看誰寫在後面，還要先看 specificity，也就是來源所說的權重。

### 畫面觀察

HTML 如下：

```html
<div class="text">你笑起來真好看</div>
```

CSS 如下：

```css
div {
    color: pink;
}

.text {
    color: red;
}
```

`div` 和 `.text` 都選到了同一個元素，也都設定 `color`。如果只看順序，`.text` 寫在後面；但真正更關鍵的是 `.text` 的權重比 `div` 高。

### 最小修改

再加入 ID 選擇器，可以看出更高權重的規則會勝出：

```html
<div class="text" id="demo">你笑起來真好看</div>
```

```css
.text {
    color: red;
}

div {
    color: pink;
}

#demo {
    color: green;
}
```

### 結果解釋

最後文字會顯示綠色，因為 `#demo` 是 ID 選擇器，權重高於類選擇器 `.text` 和標籤選擇器 `div`。

權重可以用四組數字理解：

| 類型 | 權重位置 | 範例 |
|---|---:|---|
| 行內樣式 | `(1, 0, 0, 0)` | `<p style="color: red;">` |
| ID 選擇器 | `(0, 1, 0, 0)` | `#demo` |
| 類、屬性、偽類選擇器 | `(0, 0, 1, 0)` | `.text`、`[type="text"]`、`:hover` |
| 標籤、偽元素選擇器 | `(0, 0, 0, 1)` | `div`、`p`、`::before` |

來源也用 `1000`、`100`、`10`、`1` 幫助初學者記憶。正式判斷時更建議把它看成四組位置，而不是普通十進位數字，因為權重不會跨位置進位。

### 誤判修正

不要把權重理解成單純加總後比較大小。十個類選擇器不會進位成一個 ID 選擇器；很多標籤選擇器也不會進位成一個類選擇器。

也不要把 `!important` 當成 selector 權重的一部分。來源把它放在優先級公式最右側，是為了讓初學者知道它很強。但更精確地說，`!important` 會改變 declaration 的競爭層級，而不是增加 selector 本身的 specificity。本章只保留入門判斷：不要依賴 `!important` 解決一般覆蓋問題。

### 小整理

當不同 selector 同時選到同一個元素並設定同一個屬性時，先比較 specificity。常見優先順序可以先記成：繼承值最低，通配符選擇器本身權重為 0，標籤選擇器低於類選擇器，類選擇器低於 ID 選擇器，ID 選擇器低於行內樣式；`!important` 會讓判斷變得更特殊，應謹慎使用。

## 第五課：複合選擇器的權重如何疊加

### 問題情境

實際 CSS 很少只寫單一 selector。你可能會寫 `ul li`、`.nav li` 或 `#menu .item a`。這時要判斷誰勝出，就必須把複合選擇器中每一部分的權重加起來。

### 畫面觀察

HTML 如下：

```html
<ul class="nav">
    <li>選項一</li>
    <li>選項二</li>
    <li>選項三</li>
</ul>
```

CSS 如下：

```css
ul li {
    color: green;
}

li {
    color: red;
}

.nav li {
    color: pink;
}
```

三條規則都選到了 `<li>`，而且都設定 `color`。這時要計算每條 selector 的權重。

### 最小修改

把權重寫在旁邊觀察：

```css
/* (0, 0, 0, 2)：兩個標籤選擇器 */
ul li {
    color: green;
}

/* (0, 0, 0, 1)：一個標籤選擇器 */
li {
    color: red;
}

/* (0, 0, 1, 1)：一個類選擇器 + 一個標籤選擇器 */
.nav li {
    color: pink;
}
```

### 結果解釋

最後 `<li>` 文字會顯示粉紅色，因為 `.nav li` 的權重是 `(0, 0, 1, 1)`，高於 `ul li` 的 `(0, 0, 0, 2)` 與 `li` 的 `(0, 0, 0, 1)`。

這裡最重要的是「不會進位」。即使 `ul li` 有兩個標籤選擇器，它仍然只是在最後一組數字累加成 2，不會變成類選擇器那一組。

來源中的另一個練習也能說明這件事：

```css
/* (0, 0, 0, 7) */
div div div div div div div {
    color: red;
}

/* (0, 0, 1, 0) */
.one {
    color: green;
}
```

即使第一條 selector 有七個 `div`，`.one` 仍然勝出，因為類選擇器所在的位置比標籤選擇器高。

### 誤判修正

不要用「selector 寫得越長就越強」來判斷。selector 長度不等於權重高。七個標籤選擇器仍然低於一個類選擇器；很多類選擇器仍然不會進位成 ID 選擇器。

也不要只計算 selector，忘了確認它是否真的選到元素。如果 selector 和 HTML 結構不相符，權重再高也不會影響畫面。

### 小整理

複合選擇器的權重會依照每種 selector 類型分別累加。判斷時先寫成四組數字，再由左到右比較。只要前面某一組已經分出高低，後面的數字就不需要再比。

## 第六課：父元素權重很高，為什麼子元素還是使用自己的樣式

### 問題情境

來源中有一個很容易誤判的例子：父元素使用 ID 選擇器，甚至加了 `!important`，子元素卻仍然使用自己身上的 `p` 規則。這能幫助你分清楚「直接選中」和「繼承」的差異。

### 畫面觀察

HTML 如下：

```html
<div id="father">
    <p>你還是很好看</p>
</div>
```

CSS 如下：

```css
#father {
    color: red !important;
}

p {
    color: pink;
}
```

直覺上，`#father` 權重很高，還有 `!important`，似乎應該讓 `<p>` 變紅。但畫面中的 `<p>` 會使用 `p { color: pink; }` 這條直接作用在自己身上的規則。

### 結果解釋

關鍵在於：`#father` 直接選到的是 `<div>`，不是 `<p>`。`<p>` 能拿到紅色，只是因為 `color` 可以繼承。

但是 `<p>` 自己又被 `p` selector 直接選中，並設定了 `color: pink;`。直接作用在 `<p>` 身上的 declaration，會比從父元素繼承來的值更優先。因此最後 `<p>` 顯示粉紅色。

來源把這件事整理成一句話：繼承的權重是 0。如果元素沒有被直接選中，不管父元素權重多高，子元素得到的只是繼承值。

### 誤判修正

不要把父元素的 selector 權重直接搬到子元素身上。父元素權重只用來決定父元素自己的樣式競爭結果；子元素如果只是繼承到某個值，該繼承值在子元素上的競爭力很低。

若你真的希望 `<p>` 使用紅色，可以直接選到 `<p>`：

```css
#father p {
    color: red;
}
```

這條規則的作用對象就是 `#father` 裡面的 `<p>`，不再只是依賴繼承。

### 小整理

判斷樣式時一定要先問：這條 CSS 直接選到的是哪個元素？若規則只選到父元素，子元素拿到的是繼承結果，不是父元素 selector 權重本身。

## 三大特性的整體判斷流程

學完前面幾課後，可以把三大特性整理成一個實用流程。當你要判斷某個元素最後會使用哪個屬性值時，依序問：

1. 目標元素是否被某條 CSS 規則直接選中，並設定了這個屬性？
2. 如果有多條直接規則同時設定同一個屬性，是否有 `!important` 這類特殊競爭條件？
3. 在同一競爭條件下，哪一條 selector 的 specificity 較高？
4. 如果 specificity 相同，哪一條 declaration 在 CSS 來源順序中比較後面？
5. 如果目標元素沒有直接設定這個屬性，這個屬性是否能從父元素繼承？
6. 如果不能繼承，元素會使用瀏覽器預設值或該屬性的初始值。

這個流程也能解釋本章三個名詞的關係：

| 特性 | 解決的問題 | 判斷重點 |
|---|---|---|
| 層疊性 | 多條規則衝突時，最後採用哪一條 | 同權重時看來源順序，衝突屬性後者覆蓋前者 |
| 繼承性 | 子元素沒有設定某些屬性時，是否能沿用父元素樣式 | 常見文字相關屬性可繼承，盒模型與版面屬性通常不繼承 |
| 優先級 | 不同 selector 同時選到同一元素時，誰更有競爭力 | 用 specificity 四組數字比較，權重疊加但不進位 |

這張表是回查工具。真正除錯時，仍要回到「元素是否被直接選中」和「同一屬性是否發生衝突」這兩個問題。

## 常見誤解與修正方式

### 誤解一：後寫的 CSS 一定會覆蓋前面的 CSS

後寫規則只有在同一元素、同一屬性，而且權重相同或競爭條件允許時才會勝出。如果後寫的是標籤選擇器，前面是 ID 選擇器，通常仍然是 ID 選擇器勝出。

修正方式是先比較 specificity，再看來源順序。不要一開始就只看誰寫在下面。

### 誤解二：父元素權重高，子元素就一定會跟著使用

父元素權重高，只能決定父元素自己的樣式。子元素如果只是繼承到值，這個繼承值在子元素上可以先理解為權重 0。

修正方式是確認 CSS 是否直接選到目標子元素。若要控制子元素，就寫出能匹配子元素的 selector。

### 誤解三：所有文字看起來相關的屬性都一定會繼承

來源提供的記憶方向是：`text-*`、`font-*`、`line-*` 與 `color` 常見會繼承。但實際判斷時仍要以屬性本身為準，不能只看開頭字串。

修正方式是把來源規律當成入門口訣，而不是完整規格表。遇到不確定的屬性時，應查該屬性的繼承特性。

### 誤解四：`a` 和 `h1` 是不能繼承的元素

`a` 和 `h1` 不是完全不能繼承，而是它們常常已經有瀏覽器預設樣式直接套用在自己身上，所以看起來沒有使用父元素繼承值。

修正方式是用直接 selector 覆蓋它們，例如 `div a { color: red; }` 或 `div h1 { font-size: 12px; }`。

### 誤解五：權重會像普通數字一樣進位

權重是分組比較，不是普通十進位加法。`(0, 0, 0, 10)` 不會進位成 `(0, 0, 1, 0)`。

修正方式是用四組數字由左到右比較。前面高的一方勝出，後面不用再比。

### 誤解六：selector 權重高就一定有作用

權重只在 selector 確實匹配元素時才有意義。如果 HTML 結構和 selector 不相符，這條規則根本不參與競爭。

修正方式是先確認 selector 是否選到目標元素，再計算權重。

## 實務使用情境

CSS 三大特性幾乎出現在所有樣式除錯情境中。當按鈕顏色沒有變、標題字級不符合預期、列表樣式被其他規則覆蓋，或外部樣式表和元件樣式互相影響時，都需要回到本章的判斷流程。

在寫 CSS 時，理解這三個特性能幫助你避免過度提高 selector 權重。若一開始就大量使用 ID 選擇器、很長的複合選擇器或 `!important`，後續要覆蓋樣式會變得困難。比較好的做法是讓 selector 保持必要且清楚，並利用合理的 HTML 結構、類命名與來源順序管理樣式。

在除錯時，也不要只盯著某一行 CSS。應該先確認元素是否被選中，再看同一屬性是否有其他規則競爭，最後判斷是直接指定、繼承、權重還是來源順序造成結果。

## 小結

CSS 三大特性可以放在同一條判斷線上理解：繼承性處理「元素自己沒設定時能不能沿用父元素」，優先級處理「多條不同 selector 直接選到同一元素時誰更強」，層疊性處理「條件相同時後面的衝突宣告如何覆蓋前面的宣告」。

真正判斷最終樣式時，不要只背「標籤、類、ID、行內」的順序。要先確認目標元素是否被直接選中，再比較 specificity，權重相同才看來源順序；如果沒有直接指定，才回頭看這個屬性是否能繼承。
