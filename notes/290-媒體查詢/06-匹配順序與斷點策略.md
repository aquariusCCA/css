# 匹配順序與斷點策略

## 學習目標

讀完這篇筆記，你應該能夠：

- 理解多個符合條件的 `@media` 規則會同時生效並參與層疊。
- 說明為什麼 media query 的書寫順序會影響最後結果。
- 分辨移動優先與桌面優先策略。
- 使用 `min-width` 規劃移動優先斷點。
- 使用 `max-width` 規劃桌面優先斷點。

## 問題情境

很多初學者會以為 media query 像 `if / else if` 一樣，只會命中一條。但 CSS 不是這樣運作。

如果視口寬度是 `1200px`，下面三條都可能符合：

```css
@media (min-width: 320px) {}
@media (min-width: 800px) {}
@media (min-width: 1024px) {}
```

所有符合條件的規則都會生效，最後畫面由 CSS 層疊決定。因此，media query 的順序和斷點策略非常重要。

## 一句話理解

媒體查詢負責判斷哪些規則符合條件；符合條件後，最後誰生效仍由 CSS 層疊、選擇器權重和書寫順序決定。

## 多個媒體查詢可以同時匹配

範例一：

```css
@media (min-width: 320px) {
  html {
    background: red;
  }
}

@media (min-width: 800px) {
  html {
    background: green;
  }
}

@media (min-width: 1024px) {
  html {
    background: blue;
  }
}
```

當視口寬度大於等於 `1024px` 時，三條 media query 都會匹配。

因為三段規則選擇器相同、屬性相同，且最後出現的是：

```css
background: blue;
```

所以最後背景是藍色。

## 書寫順序會影響結果

範例二：

```css
@media (min-width: 1024px) {
  html {
    background: blue;
  }
}

@media (min-width: 800px) {
  html {
    background: green;
  }
}

@media (min-width: 320px) {
  html {
    background: red;
  }
}
```

同樣在 `1024px` 以上，三條仍然都匹配。

但這次最後出現的是：

```css
background: red;
```

因此紅色會覆蓋前面的藍色和綠色。

這說明 media query 的順序不是裝飾，它會直接影響最終樣式。

## 移動優先策略

移動優先通常使用 `min-width` 規劃斷點。先寫最窄螢幕的基礎樣式，再隨著視口變寬逐步增加樣式。

![min-width 移動優先斷點策略示意圖](../../origin/290-媒體查詢/assets/images/matching-strategies-img-001-340178.webp)

```css
/* 最窄情況，通常是手機基礎樣式 */
html {
}

/* 寬度大於等於 320px */
@media (min-width: 320px) {
}

/* 寬度大於等於 1024px */
@media (min-width: 1024px) {
}
```

移動優先的思路是：

1. 先讓小螢幕可用。
2. 再為較大螢幕逐步加強布局。
3. `min-width` 從小斷點寫到大斷點，較大的規則放後面覆蓋較小的規則。

## 桌面優先策略

桌面優先通常使用 `max-width` 規劃斷點。先寫桌面樣式，再隨著視口變窄逐步覆蓋成平板或手機樣式。

![max-width 桌面優先斷點策略示意圖](../../origin/290-媒體查詢/assets/images/matching-strategies-img-002-6bd072.webp)

```css
/* 桌面基礎樣式 */
html {
}

/* 寬度小於等於 1024px */
@media (max-width: 1024px) {
}

/* 寬度小於等於 320px */
@media (max-width: 320px) {
}
```

桌面優先常見於：

- 舊專案原本就是桌面版。
- 沒有成本重構為移動優先。
- 需要在既有桌面樣式上做小螢幕降級處理。

## 移動優先與桌面優先比較

| 策略 | 常用條件 | 基礎樣式 | 後續覆蓋方向 |
| --- | --- | --- | --- |
| 移動優先 | `min-width` | 小螢幕 | 往大螢幕加樣式 |
| 桌面優先 | `max-width` | 大螢幕 | 往小螢幕覆蓋樣式 |

兩種策略沒有絕對對錯，但同一個專案最好保持一致，否則斷點順序和覆蓋關係會變得很難追蹤。

## 常見錯誤

### 錯誤一：以為只會匹配一條 media query

只要條件符合，多條 media query 可以同時生效。最後結果仍依 CSS 層疊決定。

### 錯誤二：移動優先卻把大斷點寫在前面

使用 `min-width` 時，通常應從小到大排列。

```css
@media (min-width: 320px) {}
@media (min-width: 768px) {}
@media (min-width: 1024px) {}
```

這樣較大的斷點可以自然覆蓋較小斷點。

### 錯誤三：桌面優先卻把小斷點寫在前面

使用 `max-width` 時，通常應從大到小排列。

```css
@media (max-width: 1024px) {}
@media (max-width: 768px) {}
@media (max-width: 320px) {}
```

這樣越窄的斷點越晚出現，能覆蓋較寬斷點的設定。

### 錯誤四：同一專案混用策略卻沒有規則

如果某些地方用 `min-width`，某些地方用 `max-width`，而沒有明確規範，後續維護很容易發生樣式覆蓋問題。

## 實務判斷

- 新專案或重構彈性高：可優先考慮移動優先。
- 舊桌面專案：可能採用桌面優先較符合成本。
- 使用 `min-width`：常見排列方向是小到大。
- 使用 `max-width`：常見排列方向是大到小。
- 不論採用哪種策略，都要記得：符合條件的規則都會參與層疊。

## 自我檢查

1. 寬度 `1200px` 時，`min-width: 320px`、`min-width: 800px`、`min-width: 1024px` 會匹配幾條？
2. 使用 `min-width` 做移動優先時，斷點通常應該從小到大還是從大到小？
3. 使用 `max-width` 做桌面優先時，斷點通常應該從大到小還是從小到大？
4. 如果兩條 media query 都匹配，而且選擇器權重相同、修改同一屬性，哪一條通常會勝出？
