# 瀏覽器私有前綴與相容寫法

## 學習目標

讀完這篇筆記，你應該能夠：

- 說明瀏覽器私有前綴是什麼，以及它主要出現在什麼情境。
- 辨認常見的 `-webkit-`、`-moz-`、`-ms-`、`-o-` 前綴。
- 理解為什麼歷史相容寫法通常先寫前綴版本，再寫標準版本。
- 判斷什麼時候可以移除舊的私有前綴。

## 問題情境

維護舊專案時，你可能會看到這類 CSS：

```css
-webkit-border-radius: 10px;
-moz-border-radius: 10px;
border-radius: 10px;
```

初學者很容易卡在兩個問題：這些開頭帶 `-webkit-` 或 `-moz-` 的屬性是不是新的語法？現在寫 CSS 還需不需要每次都補上這些前綴？

理解私有前綴的重點不是背下所有瀏覽器代號，而是知道它曾經用來解決什麼相容問題，以及現代專案應該如何判斷是否保留。

## 一句話理解

瀏覽器私有前綴是瀏覽器廠商在功能尚未穩定或尚未標準化時，為自己的實驗性 CSS 實作加上的識別字串。

## 私有前綴解決的是什麼問題

Web 標準通常會經過討論、實作、調整與穩定化的過程。過去當某個 CSS 特性還沒有完全穩定時，瀏覽器廠商可能會先提供自己的實驗性版本，讓開發者可以提早使用，但用私有前綴標示「這不是最終標準寫法」。

因此，私有前綴通常代表：

- 這個功能曾經是實驗性或非標準狀態。
- 不同瀏覽器可能有自己的實作差異。
- 只寫某一家瀏覽器的前綴，不能代表跨瀏覽器都會正常支援。

現代瀏覽器比較常把實驗性功能放在旗標或設定後面，不再主要依靠私有前綴來暴露新功能。不過在舊程式碼、舊教學或需要支援老版本瀏覽器的專案中，仍然可能遇到前綴寫法。

## 常見前綴

| 前綴 | 常見對應瀏覽器 |
| --- | --- |
| `-webkit-` | WebKit / Blink 系瀏覽器，例如 Safari、Chrome、Chromium 系瀏覽器與 iOS 瀏覽器 |
| `-moz-` | Firefox |
| `-ms-` | Internet Explorer 與舊版 Edge |
| `-o-` | 舊版 Presto Opera |

這張表適合用來讀懂舊 CSS，但不代表寫新專案時要手動把所有前綴都補上。

## 前綴與標準寫法的順序

如果專案確實需要支援舊瀏覽器，而且某個屬性仍需要保留前綴，常見寫法是先寫帶前綴的版本，最後再寫標準版本：

```css
.card {
  -moz-border-radius: 10px;
  -webkit-border-radius: 10px;
  -o-border-radius: 10px;
  border-radius: 10px;
}
```

這裡的 `border-radius` 是歷史相容範例。現代瀏覽器通常已經不需要為 `border-radius` 手動加前綴。

## 為什麼標準寫法要放最後

CSS 會依照來源順序套用規則：當同一個屬性有多個可識別的宣告時，後面的宣告會覆蓋前面的宣告。

把標準屬性放最後有兩個好處：

- 支援標準屬性的瀏覽器會採用最後的標準寫法。
- 只支援舊前綴的瀏覽器仍有機會讀到它認得的前綴版本。

也就是說，前綴版本是給舊實作的退路，標準版本才是希望瀏覽器優先採用的正式寫法。

## 什麼時候可以移除私有前綴

當專案需要支援的瀏覽器版本都已支援標準寫法，就可以移除對應的私有前綴。實務上通常不會靠人工逐一記憶，而是交給工具或支援清單判斷。

常見判斷方式包括：

- 專案的 Browserslist 已不再包含需要該前綴的舊瀏覽器版本。
- Autoprefixer 判定該屬性不需要再產生前綴。
- 目標瀏覽器都已穩定支援不帶前綴的標準屬性。

如果沒有明確支援舊瀏覽器的需求，新寫 CSS 時通常應優先使用標準寫法。

## 實務使用建議

在現代專案中，建議把前綴管理交給工具，而不是手動背每個屬性在每個瀏覽器的歷史支援狀態。

比較好的工作方式是：

1. 在原始 CSS 中撰寫標準屬性。
2. 用 Browserslist 描述專案需要支援的瀏覽器範圍。
3. 讓 Autoprefixer 依支援範圍自動補上必要前綴。
4. 當支援範圍更新時，讓工具移除不再需要的前綴。

這樣能避免兩種極端：完全不處理舊瀏覽器相容性，或永遠保留大量已不需要的歷史前綴。

## 常見錯誤

### 只寫前綴版本，漏掉標準版本

```css
.card {
  -webkit-border-radius: 10px;
}
```

這種寫法只照顧到特定瀏覽器的舊實作。當瀏覽器支援標準屬性時，應該補上不帶前綴的版本：

```css
.card {
  -webkit-border-radius: 10px;
  border-radius: 10px;
}
```

### 把舊前綴當成現代 CSS 必備語法

看到舊程式碼有前綴，不代表現在每個新屬性都要自己手動加前綴。很多常見 CSS3 特性在主流瀏覽器中已穩定支援，手動加入舊前綴反而會讓樣式表變得雜亂。

### 不看支援目標就刪除所有前綴

如果專案仍明確支援舊瀏覽器，貿然刪除前綴可能造成樣式失效。移除前綴前，應先確認支援範圍，或讓 Autoprefixer 根據 Browserslist 重新產生結果。

## 重點整理

- 私有前綴是瀏覽器廠商過去用來標示實驗性、非標準或尚未穩定 CSS 實作的寫法。
- 常見前綴包含 `-webkit-`、`-moz-`、`-ms-`、`-o-`。
- 如果需要保留前綴，通常先寫前綴版本，最後寫標準版本。
- 現代開發應優先撰寫標準屬性，並用 Browserslist / Autoprefixer 判斷是否需要補前綴。
- 可以移除前綴的前提是：專案目標瀏覽器都已支援標準寫法，或工具判定不再需要。

## 自我檢查

1. 為什麼只寫 `-webkit-` 版本不等於完成跨瀏覽器支援？
2. 在保留前綴與標準屬性時，為什麼通常要把標準屬性放在最後？
3. 如果一個專案使用 Autoprefixer，你是否還需要手動記住每個 CSS 屬性該加哪些前綴？
4. 移除私有前綴前，應該先確認哪兩件事？
