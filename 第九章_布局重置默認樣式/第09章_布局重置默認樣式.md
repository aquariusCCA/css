## 這一章先抓結論

1. 瀏覽器對元素本來就有一套默認樣式。
2. 如果不先處理這些默認樣式，版面會很難精準控制。
3. 小型示範可以用簡單重置，大型專案更適合選擇較完整的 reset 或 normalize 方案。

## 30 秒複習入口

- 想把所有元素先清乾淨，再自己重做版面，常用 `reset.css`
- 想保留元素合理的預設外觀，只修正不同瀏覽器的差異，常用 `normalize.css`
- 想在簡單案例快速消除邊距，可用 `* { margin: 0; padding: 0; }`

---

## 1. 為什麼要重置默認樣式

不同瀏覽器對標籤的預設樣式不完全一樣。  
這些默認樣式在簡單頁面裡問題不大，但在正式排版時，常會造成以下麻煩：

- 版面左右上下多出意料之外的空白
- 標題、段落、清單的間距不一致
- 圖片、按鈕、表單元件在不同瀏覽器顯示不同
- 相同 HTML 結構，換瀏覽器就出現不同結果

常見的默認樣式例子：

- `p` 元素通常有上下 `margin`
- `h1 ~ h6` 元素通常有上下 `margin`，而且字體較粗
- `body` 通常有預設的 `margin`

所以在開始做版面前，先處理默認樣式是很常見的做法。

---

## 2. 方案一：使用全局選擇器快速重置

最簡單的做法是用全局選擇器把常見空白清掉：

```css
* {
  margin: 0;
  padding: 0;
}
```

### 這種做法的特點

- 寫法最快
- 適合簡單示範或小型練習
- 可以快速消掉大部分元素的預設外距與內距

### 為什麼不建議直接當成正式方案

- `*` 會選到所有元素
- 並不是所有元素都需要被同樣處理
- 有些元素的預設樣式本來就有意義，不一定要全部清掉
- 後續如果要做特殊元素的樣式，可能會和這種「全清空」思路衝突

### 什麼時候可以用

- 練習頁面
- 簡單 Demo
- 臨時測試版面

---

## 3. 方案二：reset.css

`reset.css` 的思路是：

> 把具有默認樣式效果的元素，先清成接近白紙的狀態，再由開發者依設計稿重新建立樣式。

這種方式比單純 `* { margin: 0; padding: 0; }` 更完整，通常會一起處理：

- 外邊距和內邊距
- 斜體、粗體等文字樣式
- 清單圓點
- 圖片邊框與對齊問題
- 按鈕滑鼠游標
- 連結顏色與底線
- `body` 的字型、字級、行高與底色
- 常見的清浮動工具類

### 常見的 reset 規則

```css
* {
  margin: 0;
  padding: 0;
}

em,
i {
  font-style: normal;
}

li {
  list-style: none;
}

img {
  border: 0;
  vertical-align: middle;
}

button {
  cursor: pointer;
}

a {
  color: #666;
  text-decoration: none;
}

a:hover {
  color: #c81623;
}

body {
  background-color: #fff;
  font: 12px/1.5 Microsoft YaHei, Heiti SC, tahoma, arial, Hiragino Sans GB, "\5B8B\4F53", sans-serif;
  color: #666;
  -webkit-font-smoothing: antialiased;
}

.hide,
.none {
  display: none;
}

.clearfix:after {
  visibility: hidden;
  clear: both;
  display: block;
  content: ".";
  height: 0;
}

.clearfix {
  *zoom: 1;
}
```

### 這些規則在做什麼

- `* { margin: 0; padding: 0; }`
  - 把常見元素的外距、內距清掉
- `em, i { font-style: normal; }`
  - 取消斜體預設
- `li { list-style: none; }`
  - 去掉清單圓點
- `img { border: 0; vertical-align: middle; }`
  - 避免圖片邊框與底部空白縫隙問題
- `button { cursor: pointer; }`
  - 讓按鈕有明確互動感
- `a { text-decoration: none; }`
  - 去掉底線，方便自己重新定義連結樣式
- `body { ... }`
  - 建立整站共用的字型、字級與底色
- `.clearfix`
  - 處理浮動佈局的收尾問題

### 什麼時候用

- 要從一張比較乾淨的白紙開始做版面
- 專案會自己完整定義元件外觀
- 想讓同一份樣式在多數頁面有一致起點

### 使用提醒

- `reset.css` 通常會清得比較徹底
- 清得越徹底，後面自己補樣式的工作就越多
- 如果專案已經有統一規範，就不要再混入多套不同的 reset

---

## 4. 方案三：Normalize.css

`normalize.css` 的思路和 `reset.css` 不一樣。  
它不是把所有東西都清成白紙，而是：

- 保留有意義的預設值
- 修正不同瀏覽器之間不一致的地方
- 讓各瀏覽器的表現更接近

### 這種做法的重點

- 比起全面清空，更重視跨瀏覽器一致性
- 對某些元素保留原本語義和可用性
- 比較適合想要「統一差異」而不是「全部推倒重來」的情境

### Normalize.css 的使用方式

一般有兩種常見做法：

1. 把 `normalize.css` 當成專案基礎樣式之一，再接自己的客製樣式
2. 引入 `normalize.css` 原始碼，在其上方或下方疊加自己的規則

### 什麼時候用

- 想減少瀏覽器差異，但不想過度清空預設樣式
- 想保留元素合理的基本外觀
- 專案需要比較穩定、可維護的樣式起點

---

## 5. `reset.css` 和 `normalize.css` 怎麼選

| 項目 | `reset.css` | `normalize.css` |
| --- | --- | --- |
| 核心目的 | 盡量把默認樣式清掉 | 修正瀏覽器差異，保留合理預設 |
| 風格 | 比較激進 | 比較保守 |
| 結果 | 接近白紙 | 接近統一標準 |
| 適合情境 | 自己會重做大部分樣式的專案 | 想保留一些合理預設、減少差異的專案 |

### 簡單判斷

- 如果你想要「從頭自己畫」
  - 偏向 `reset.css`
- 如果你想要「先統一瀏覽器差異，再微調」
  - 偏向 `normalize.css`

---

## 6. 易錯點

- 不是所有默認樣式都一定是錯的，有些只是方便使用者操作。
- `* { margin: 0; padding: 0; }` 只是快速重置，不等於正式專案最佳解。
- `reset.css` 和 `normalize.css` 不是同一種思路，不要混為一談。
- 重置樣式不是目的，目的是讓後續版面控制更穩定。

## 7. 一句話抓核心

先處理瀏覽器默認樣式，才能把版面控制建立在一致的起點上；要全面清空用 `reset.css`，要保留合理預設用 `normalize.css`。
