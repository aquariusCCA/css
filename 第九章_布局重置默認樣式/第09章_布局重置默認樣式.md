## 這一章先抓結論

1. 瀏覽器本來就會替 HTML 元素套上一套預設樣式，這些樣式通常來自瀏覽器的 **User Agent Stylesheet**。
2. 不同瀏覽器的預設樣式可能不完全一致，所以正式排版前通常會先做重置或標準化。
3. `* { margin: 0; padding: 0; }` 是最快的練習型重置，但不是完整工程方案。
4. `reset.css` 的思路是「把預設樣式清得更乾淨，再由專案自己重建樣式」。
5. `normalize.css` 的思路是「保留合理預設，只修正不同瀏覽器之間的不一致」。
6. 現代 CSS reset 通常會加入 `box-sizing: border-box`、媒體元素自適應、表單字型繼承等規則。
7. 使用組件庫時，reset 應該保守，避免全局規則破壞組件內部樣式。

- 想保留合理預設，只統一瀏覽器差異：偏向 `normalize.css`

# 1. 為什麼瀏覽器會有默認樣式

HTML 元素不是完全沒有樣式的。即使你沒有寫任何 CSS，瀏覽器也會替不同標籤套上預設樣式。

這些預設樣式通常包含：

- `body` 預設有 `margin`
- `p` 預設有上下外距
- `h1 ~ h6` 預設有字級、粗體、上下外距
- `ul`、`ol` 預設有縮排與清單符號
- `a` 預設有顏色與底線
- `button`、`input`、`select`、`textarea` 有瀏覽器原生外觀
- `img` 是 inline 元素，底部可能會出現空隙

這些預設樣式不是錯誤。它們的存在是為了讓沒有 CSS 的 HTML 文件仍然可以閱讀、操作、呈現基本層次。

但是在正式開發中，問題會出現在：

```html
<h1>標題</h1>
<p>段落內容</p>
```

你可能以為它們只會照著 HTML 順序排列，但實際上瀏覽器已經幫它們加上字級、粗體、上下外距。當你開始切版時，這些預設值可能會讓畫面出現你沒有預期的空白。

---

# 2. 默認樣式會造成哪些工程問題

## 2.1 版面空白不容易控制

最常見的問題是：畫面上多出不明原因的空白。

例如：

```html
<body>
  <div class="app">內容</div>
</body>
```

即使 `.app` 沒有設定 margin，畫面邊緣仍可能有空白，原因通常是 `body` 本身有預設 `margin`。

---

## 2.2 標題、段落、清單間距不一致

`h1`、`p`、`ul`、`ol` 都有自己的預設外距。

如果你在做卡片、表單、後台列表、彈窗內容時沒有先處理，常會看到：

- 標題往上或往下撐出空白
- 段落之間距離不受控
- 清單左邊自動縮排
- 同一份 HTML 在不同瀏覽器看起來略有差異

---

## 2.3 表單元素外觀差異明顯

表單元素是瀏覽器差異最明顯的地方之一。

例如：

- `button` 的內距、邊框、背景不同
- `input` 的字型不一定繼承 `body`
- `select` 在不同作業系統和瀏覽器上外觀差異很大
- `textarea` 可能有不同的預設 resize 行為

所以很多工程專案會對表單元素做基本統一，但不一定會完全清空。

---

## 2.4 組件庫可能被全局 reset 影響

如果專案使用 View UI Plus、Element Plus、Ant Design Vue 這類組件庫，組件庫本身已經定義好按鈕、輸入框、表格、彈窗、下拉選單等樣式。

如果你寫了太激進的全局 reset，例如：

```css
button,
input,
select,
textarea {
  border: none;
  outline: none;
  background: none;
}
```

就可能破壞組件庫的按鈕、輸入框、下拉選單或可訪問性狀態。

所以在有組件庫的專案中，reset 要更保守。

# 3. 方案一：使用全局選擇器快速重置

最簡單的寫法是：

```css
* {
  margin: 0;
  padding: 0;
}
```

這段程式碼的意思是：把所有元素的外距與內距先歸零。

---

## 3.1 它解決了什麼

它主要解決：

- `body` 的預設 margin
- `p` 的預設上下 margin
- `h1 ~ h6` 的預設上下 margin
- `ul`、`ol` 的預設 padding
- 大部分元素造成的預設空白

對於初學切版、小型 Demo、臨時測試，它很直覺，也很有效。

---

## 3.2 它沒有解決什麼

但它不是完整方案，因為它沒有處理：

- `box-sizing` 盒模型
- 圖片底部空隙
- 圖片、影片是否能自適應容器
- 表單元素字型是否繼承
- 連結樣式
- 清單符號是否要保留
- 可訪問性焦點樣式
- 不同瀏覽器對元素的細節差異

所以它只能算「快速清空外距與內距」，不能算完整的 CSS 初始化方案。

---

## 3.3 為什麼正式專案不建議只靠它

因為 `*` 會選到所有元素，而且它只做兩件事：清 margin、清 padding。

正式專案通常需要的是：

- 建立一致盒模型
- 處理圖片與媒體元素
- 處理表單元素
- 保留必要的可用性與可訪問性
- 避免破壞第三方組件庫

所以正式專案通常會改成「小型 reset + base.css + 專案規範」的方式。

---

# 4. 方案二：reset.css

`reset.css` 的核心思路是：

> 把具有預設樣式效果的元素，先清成接近白紙的狀態，再由開發者依照設計稿重新建立樣式。

它比單純的 `* { margin: 0; padding: 0; }` 更完整，也更激進。

---

## 4.1 reset.css 常處理哪些內容

常見的 reset 會處理：

- 清除常見元素的 `margin`、`padding`
- 清除清單符號
- 清除連結底線與顏色
- 統一 `body` 字型、字級、行高、背景色
- 統一圖片顯示方式
- 統一按鈕滑鼠游標
- 清除 `em`、`i` 的斜體
- 提供 `.clearfix` 工具類
- 處理舊瀏覽器兼容性

---

## 4.2 傳統 reset.css 範例

```css
* {
  margin: 0;
  padding: 0;
}

em,
i {
  font-style: normal;
}

li {
  list-style: none;
}

img {
  border: 0;
  vertical-align: middle;
}

button {
  cursor: pointer;
}

a {
  color: #666;
  text-decoration: none;
}

a:hover {
  color: #c81623;
}

body {
  background-color: #fff;
  font: 12px/1.5 Microsoft YaHei, Heiti SC, tahoma, arial, Hiragino Sans GB, "\5B8B\4F53", sans-serif;
  color: #666;
  -webkit-font-smoothing: antialiased;
}

.hide,
.none {
  display: none;
}

.clearfix::after {
  visibility: hidden;
  clear: both;
  display: block;
  content: ".";
  height: 0;
}

.clearfix {
  *zoom: 1;
}
```

---

## 4.3 這些規則分別在做什麼

| 規則 | 目的 | 注意事項 |
| --- | --- | --- |
| `* { margin: 0; padding: 0; }` | 清除預設外距與內距 | 適合快速建立乾淨起點，但不夠完整 |
| `em, i { font-style: normal; }` | 取消斜體 | 如果內容語義需要強調，不一定要取消 |
| `li { list-style: none; }` | 去掉清單圓點 | 文章內容區通常不建議全部移除 |
| `img { border: 0; vertical-align: middle; }` | 避免圖片邊框與底部空隙 | 現代專案更常用 `display: block; max-width: 100%;` |
| `button { cursor: pointer; }` | 讓按鈕有互動感 | disabled 狀態要另外處理 |
| `a { text-decoration: none; }` | 去掉連結底線 | 內容型網站可能不建議全局移除 |
| `body { font: ... }` | 設定整站基礎字型 | 現代專案通常使用 `system-ui` 或設計系統字型 |
| `.clearfix` | 清除浮動影響 | Flex/Grid 普及後使用頻率降低 |

---

## 4.4 reset.css 適合什麼情境

適合：

- 從設計稿完整重做視覺樣式
- 公司有自己的設計系統
- 頁面大多由自製元件組成
- 專案希望所有元素從一致起點開始

不太適合：

- 文章型網站，因為內容需要保留標題、段落、清單的可讀性
- 大量依賴第三方組件庫的專案
- 不清楚每條 reset 作用就整包貼上的情況

---

# 5. 方案三：Normalize.css

`normalize.css` 的核心思路和 `reset.css` 不同。

它不是把所有元素清成白紙，而是：

1. 保留有意義的預設值
2. 修正不同瀏覽器的不一致
3. 讓各瀏覽器的表現更接近
4. 維持元素基本可用性與可讀性

---

## 5.1 Normalize.css 的精神

可以這樣理解：

- `reset.css`：我不要瀏覽器幫我設計，我全部自己來。
- `normalize.css`：瀏覽器合理的部分留下，不一致的地方幫我修正。

所以 `normalize.css` 比較保守，也比較重視跨瀏覽器一致性。

---

## 5.2 Normalize.css 適合什麼情境

適合：

- 想保留標題、段落、表單等合理預設
- 想減少瀏覽器差異，但不想全部推倒重來
- 專案中有大量文字內容
- 不想因為 reset 太激進而補回大量基礎樣式
- 使用組件庫，希望全局樣式保持保守

---

## 5.3 使用方式

常見方式有兩種。

### 方式一：透過 npm 安裝

```bash
npm install normalize.css
```

然後在入口檔引入：

```ts
// main.ts
import 'normalize.css'
import './assets/styles/base.css'
import './assets/styles/index.css'
```

### 方式二：複製需要的規則到專案 reset.css

如果專案不想額外依賴，也可以挑出需要的 normalize 規則，放進自己的 `reset.css` 或 `base.css`。

但要注意：不要只複製看起來熟悉的規則，而是要知道每條規則解決什麼問題。

---

# 6. 方案四：現代 CSS Reset

現代前端專案通常不一定使用傳統大型 reset.css，而是使用較精簡的 modern reset。

它的核心目標是：

- 修正最常見的版面問題
- 建立合理盒模型
- 讓媒體元素自適應容器
- 讓表單元素繼承字型
- 保留必要的可訪問性
- 避免過度清空瀏覽器預設

---

## 6.1 現代 reset 常見範例

```css
/* 1. 使用更直覺的盒模型 */
*,
*::before,
*::after {
  box-sizing: border-box;
}

/* 2. 移除 body 預設 margin，建立基礎排版 */
body {
  margin: 0;
  min-height: 100vh;
  line-height: 1.5;
  font-family: system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif;
  -webkit-font-smoothing: antialiased;
}

/* 3. 避免標題與段落預設 margin 影響版面 */
h1,
h2,
h3,
h4,
h5,
h6,
p {
  margin: 0;
}

/* 4. 圖片與媒體元素預設自適應容器 */
img,
picture,
video,
canvas,
svg {
  display: block;
  max-width: 100%;
}

/* 5. 表單元素繼承字型 */
input,
button,
textarea,
select {
  font: inherit;
}

/* 6. 按鈕保留互動提示 */
button {
  cursor: pointer;
}

button:disabled {
  cursor: not-allowed;
}

/* 7. 保留鍵盤操作時的可見焦點 */
:focus-visible {
  outline: 2px solid currentColor;
  outline-offset: 2px;
}

/* 8. 尊重使用者減少動畫的系統設定 */
@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    scroll-behavior: auto !important;
    transition-duration: 0.01ms !important;
  }
}
```

---

## 6.2 為什麼要加 `box-sizing: border-box`

預設盒模型是 `content-box`。

也就是說：

```css
.card {
  width: 300px;
  padding: 20px;
  border: 1px solid #ccc;
}
```

如果使用預設的 `content-box`，實際寬度會變成：

```text
300px + 20px * 2 + 1px * 2 = 342px
```

這很容易造成版面超出容器。

改成：

```css
*,
*::before,
*::after {
  box-sizing: border-box;
}
```

之後，`width: 300px` 就會包含 content、padding、border，實際排版更直覺。

---

## 6.3 為什麼圖片要設定 `display: block`

`img` 預設是 inline 元素，會受到文字基線影響，底部可能出現一點空隙。

常見處理：

```css
img {
  display: block;
  max-width: 100%;
}
```

效果是：

- `display: block`：避免 inline 圖片底部空隙
- `max-width: 100%`：避免圖片超出父容器

---

## 6.4 為什麼表單元素要 `font: inherit`

表單元素在部分瀏覽器中不一定完全繼承父層字型。

所以常見寫法是：

```css
input,
button,
textarea,
select {
  font: inherit;
}
```

這樣表單元素會和頁面文字保持一致。

但注意：這不代表要把它們的 `border`、`background`、`outline` 全部清掉。

---

# 7. `reset.css`、`normalize.css`、modern reset 怎麼選

| 方案 | 核心目的 | 清除程度 | 優點 | 風險 | 適合情境 |
| --- | --- | --- | --- | --- | --- |
| `*` 快速重置 | 快速清掉 margin/padding | 低 | 簡單、好懂、適合練習 | 不完整 | Demo、小練習 |
| `reset.css` | 清成接近白紙 | 高 | 起點乾淨、完全可控 | 後續要補大量樣式，可能破壞組件庫 | 自製設計系統、強視覺客製專案 |
| `normalize.css` | 修正瀏覽器差異 | 中低 | 保留合理預設、穩定 | 不會幫你清掉所有版面差異 | 內容型網站、一般專案、組件庫專案 |
| modern reset | 解決現代常見問題 | 中 | 精簡、實用、容易維護 | 需要自己判斷要保留哪些規則 | Vue / React / Vite 現代專案 |

---

## 7.1 簡單決策流程

### 情境一：只是練習切版

用：

```css
* {
  margin: 0;
  padding: 0;
}
```

理由：快速看到效果，不需要過度設計。

---

### 情境二：自己做完整 UI，不依賴組件庫

可以用：

```text
reset.css + base.css + components.css
```

理由：你會自己定義按鈕、表單、卡片、表格、彈窗，所以起點乾淨是好事。

---

### 情境三：專案有大量文章內容

建議用：

```text
normalize.css + typography.css
```

理由：文章內容通常需要保留標題、段落、清單、連結的可讀性，不適合全部清空。

---

### 情境四：Vue 專案使用 View UI Plus / Element Plus

建議用：

```text
保守 modern reset + 組件庫樣式 + 少量專案覆蓋
```

理由：組件庫本身已經有完整樣式，過度 reset 容易造成副作用。

---

# 8. Vue / Vite 專案中怎麼放 reset.css

在 Vue / Vite 專案中，reset 通常是全局樣式，不應該寫在某個元件的 `<style scoped>` 裡。

推薦結構：

```text
src/
  assets/
    styles/
      reset.css       # 重置或標準化樣式
      variables.css   # CSS 變數、色彩、間距、字級
      base.css        # body、html、全站基礎排版
      utilities.css   # 工具類，可選
      index.css       # 統一匯入入口
```

---

## 8.1 index.css 統一匯入

```css
/* src/assets/styles/index.css */
@import './reset.css';
@import './variables.css';
@import './base.css';
@import './utilities.css';
```

然後在 `main.ts` 引入：

```ts
// main.ts
import { createApp } from 'vue'
import App from './App.vue'

import './assets/styles/index.css'

createApp(App).mount('#app')
```

這種方式的好處是：入口清楚，所有全局樣式由 `index.css` 管理。

---

## 8.2 使用組件庫時的引入順序

如果使用 View UI Plus 這類組件庫，可以用類似順序：

```ts
// main.ts
import { createApp } from 'vue'
import ViewUIPlus from 'view-ui-plus'
import App from './App.vue'

// 1. 先放專案自己的基礎 reset
import './assets/styles/reset.css'
import './assets/styles/base.css'

// 2. 再放組件庫樣式，避免 reset 蓋掉組件庫內部樣式
import 'view-ui-plus/dist/styles/viewuiplus.css'

// 3. 最後才放少量、明確的專案覆蓋樣式
import './assets/styles/index.css'

const app = createApp(App)
app.use(ViewUIPlus)
app.mount('#app')
```

重點是：

- reset 要保守
- 組件庫樣式不要被粗暴覆蓋
- 真正要改組件庫外觀時，用明確的 class 或主題變數處理
- 不要用全局 `button`、`input`、`.ivu-*` 大範圍亂改

---

## 8.3 不建議這樣做

不建議把 reset 寫在某個元件裡：

```vue
<style scoped>
* {
  margin: 0;
  padding: 0;
}
</style>
```

原因：

1. `scoped` 會讓樣式只作用於該元件範圍，不是真正的全局 reset。
2. 每個元件都寫 reset 會造成重複與維護困難。
3. 全局基礎規則應該集中管理，避免散落在各個元件。

---

# 9. 使用組件庫時的 reset 注意事項

如果專案使用 View UI Plus、Element Plus、Ant Design Vue，reset 要更謹慎。

---

## 9.1 可以安全保留的全局規則

通常比較安全：

```css
*,
*::before,
*::after {
  box-sizing: border-box;
}

body {
  margin: 0;
  min-height: 100vh;
}

img,
svg,
video,
canvas {
  display: block;
  max-width: 100%;
}

input,
button,
textarea,
select {
  font: inherit;
}
```

這些規則偏向基礎排版，不太會破壞組件庫的完整外觀。

---

## 9.2 要小心的全局規則

下面這些規則要小心：

```css
button,
input,
select,
textarea {
  border: none;
  outline: none;
  background: none;
}
```

風險：

- 可能讓組件庫按鈕失去邊框與背景
- 可能讓輸入框 focus 狀態消失
- 可能破壞 disabled、error、warning 等狀態樣式
- 可能降低鍵盤操作可見性

也要小心：

```css
a {
  color: inherit;
  text-decoration: none;
}

ul,
ol {
  list-style: none;
  padding: 0;
}
```

這些規則不是不能用，而是要看專案類型。

如果是後台系統，清掉連結底線和清單符號通常可以接受。  
如果是文章內容、文件系統、CMS 頁面，清掉它們反而會降低可讀性。

---

## 9.3 建議用低權重 reset

如果想降低 reset 對元件樣式的干擾，可以使用 `:where()`：

```css
:where(h1, h2, h3, h4, h5, h6, p) {
  margin: 0;
}
```

`where()` 的選擇器權重是 0，後續元件樣式比較容易覆蓋它。

這種寫法適合基礎 reset，因為 reset 本來就不應該比元件樣式更強。

---

# 10. 推薦的專案 reset 範本

以下是一份比較適合 Vue / Vite / 一般後台系統的保守版 reset。

它不追求把所有預設樣式清乾淨，而是處理最常見的工程問題。

```css
/* src/assets/styles/reset.css */

*,
*::before,
*::after {
  box-sizing: border-box;
}

html {
  -webkit-text-size-adjust: 100%;
}

body {
  margin: 0;
  min-height: 100vh;
  line-height: 1.5;
  font-family: system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif;
  color: #1f2937;
  background-color: #fff;
  -webkit-font-smoothing: antialiased;
  -moz-osx-font-smoothing: grayscale;
}

:where(h1, h2, h3, h4, h5, h6, p) {
  margin: 0;
}

:where(img, picture, video, canvas, svg) {
  display: block;
  max-width: 100%;
}

:where(input, button, textarea, select) {
  font: inherit;
}

:where(button) {
  cursor: pointer;
}

:where(button:disabled) {
  cursor: not-allowed;
}

:where(a) {
  color: inherit;
}

:where(code, kbd, samp, pre) {
  font-family: ui-monospace, SFMono-Regular, Menlo, Monaco, Consolas, "Liberation Mono", "Courier New", monospace;
}

:focus-visible {
  outline: 2px solid currentColor;
  outline-offset: 2px;
}

@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    scroll-behavior: auto !important;
    transition-duration: 0.01ms !important;
  }
}
```

---

## 10.1 為什麼這份 reset 比較保守

它沒有做這些事：

```css
button,
input {
  border: none;
  outline: none;
  background: none;
}
```

也沒有直接做：

```css
a {
  text-decoration: none;
}

ul,
ol {
  list-style: none;
}
```

原因是這些規則會改變語義元素的可讀性與互動提示，而且比較容易影響組件庫。

如果你的專案確定是後台管理系統，並且清單與連結都由自製元件控制，才可以把它們加進去。

# 11. reset.css 和 base.css 的分工

很多初學者會把所有全局樣式都塞進 `reset.css`，但工程上更建議拆開。

---

## 11.1 reset.css 負責什麼

`reset.css` 負責處理瀏覽器預設差異，例如：

- `box-sizing`
- `body margin`
- 圖片自適應
- 表單字型繼承
- 基礎 focus 樣式

重點是：讓環境變得穩定。

---

## 11.2 base.css 負責什麼

`base.css` 負責專案自己的基礎規範，例如：

```css
/* src/assets/styles/base.css */
html,
body,
#app {
  width: 100%;
  min-height: 100%;
}

body {
  font-size: 14px;
  color: var(--color-text-primary);
  background-color: var(--color-bg-page);
}

.page-container {
  padding: 24px;
}
```

重點是：定義專案自己的視覺基礎。

---

## 11.3 variables.css 負責什麼

`variables.css` 可以放設計變數：

```css
:root {
  --color-text-primary: #1f2937;
  --color-text-secondary: #6b7280;
  --color-bg-page: #f5f7fa;
  --color-bg-card: #ffffff;

  --space-xs: 4px;
  --space-sm: 8px;
  --space-md: 16px;
  --space-lg: 24px;

  --radius-sm: 4px;
  --radius-md: 8px;
  --radius-lg: 12px;
}
```

這樣 reset、base、design token 的責任會比較清楚。

---

# 12. 常見易錯點

## 12.1 以為默認樣式都是錯的

不是。

瀏覽器預設樣式本來是為了可讀性與可用性存在的。重置不是為了消滅所有預設，而是為了讓專案更可控。

---

## 12.2 把 `* { margin: 0; padding: 0; }` 當成完整 reset

它只是快速清除外距與內距，沒有處理盒模型、媒體元素、表單、可訪問性與跨瀏覽器差異。

---

## 12.3 reset 寫得太激進

例如：

```css
* {
  margin: 0;
  padding: 0;
  border: 0;
  outline: 0;
}
```

這種寫法可能讓鍵盤使用者看不到 focus 狀態，也可能破壞表單和組件庫樣式。

---

## 12.4 在 scoped style 裡寫全局 reset

reset 是全局基礎樣式，應該集中放在全局 CSS 檔案。

不要分散在多個元件裡。

---

## 12.5 專案已經有組件庫，還自己大範圍覆蓋原生元素

組件庫通常依賴自己的 class、狀態 class、CSS 變數或 Less/Sass 變數。

要改組件庫外觀時，優先使用：

- 官方主題配置
- CSS 變數
- 指定元件 class
- 局部 wrapper class

不要用全局元素選擇器粗暴覆蓋。

---

# 13. 面試可以怎麼回答

如果面試官問：**為什麼需要 reset.css 或 normalize.css？**

可以回答：

> 因為瀏覽器會替 HTML 元素套用預設樣式，而且不同瀏覽器的預設樣式可能不一致。為了讓版面控制建立在一致的起點上，專案通常會先做樣式重置或標準化。`reset.css` 偏向把預設樣式清乾淨，再由專案自己重建；`normalize.css` 則偏向保留合理預設，只修正瀏覽器差異。現代專案通常會採用較精簡的 reset，例如設定 `box-sizing: border-box`、移除 `body margin`、讓圖片自適應、讓表單元素繼承字型。若專案使用組件庫，reset 會更保守，避免破壞組件庫內部樣式。

---

# 14. 本章工程實戰檢查清單

建立專案 reset 時，可以檢查：

- [ ] 是否有設定 `box-sizing: border-box`
- [ ] 是否有移除 `body` 預設 `margin`
- [ ] 是否有設定基礎 `line-height`
- [ ] 圖片、影片、SVG 是否會超出容器
- [ ] 表單元素是否繼承字型
- [ ] 是否保留鍵盤 focus 樣式
- [ ] 是否避免全局移除 `outline`
- [ ] 是否避免過度重置 `button`、`input`、`select`、`textarea`
- [ ] 使用組件庫時，reset 是否足夠保守
- [ ] 是否把 reset.css、base.css、variables.css 分工清楚
- [ ] 是否在 `main.ts` 或全局入口中引入，而不是寫在 scoped style 裡

---

# 15. 一句話抓核心

重置樣式不是為了把瀏覽器預設全部消滅，而是為了建立一致、可預期、可維護的樣式起點；小練習可以用 `*` 快速重置，完整客製專案可用 `reset.css`，想保留合理預設可用 `normalize.css`，現代 Vue / Vite 專案則更適合採用保守的 modern reset。
