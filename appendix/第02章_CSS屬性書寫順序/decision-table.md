# 問題對應檢查表

來源筆記：`notes/第02章_CSS屬性書寫順序/CSS屬性書寫順序.md`

用途：遇到閱讀、維護或畫面問題時，快速判斷應優先檢查哪個 CSS 屬性群組或條件。

| 畫面問題 | 優先檢查 | 建議檢查或語法 | 判斷理由 | 來源 |
|---|---|---|---|---|
| 一段 CSS 可以生效，但很難閱讀 | declaration 是否依固定順序分組 | 布局定位 → 自身屬性 → 文本屬性 → 其他屬性 | 問題不是瀏覽器看不懂，而是人很難快速找到修改位置 | [先看問題](../../notes/第02章_CSS屬性書寫順序/CSS屬性書寫順序.md#先看問題一段-css-能生效但很難閱讀) |
| 不知道讀一個 selector 時要先看哪條宣告 | 是否先看布局定位區段 | `display`、`position`、`float`、`clear`、`visibility`、`overflow` | 先建立元素如何參與版面的判斷 | [第一課](../../notes/第02章_CSS屬性書寫順序/CSS屬性書寫順序.md#第一課先寫布局定位先判斷元素如何參與版面) |
| 元素位置或排列不對 | 布局定位屬性是否集中且靠前 | `display`、`position`、`float`、`clear` | 位置與排列問題通常應先確認版面角色 | [實務使用情境](../../notes/第02章_CSS屬性書寫順序/CSS屬性書寫順序.md#實務使用情境) |
| 內容超出或被裁切 | 是否有溢出處理設定 | `overflow` | 筆記把 `overflow` 放在布局定位屬性，用於先檢查超出如何處理 | [第一課](../../notes/第02章_CSS屬性書寫順序/CSS屬性書寫順序.md#第一課先寫布局定位先判斷元素如何參與版面) |
| 尺寸或間距不對 | 自身屬性區段是否完整 | `width`、`height`、`margin`、`padding`、`border` | 這些屬性描述元素盒子的大小與空間 | [第二課](../../notes/第02章_CSS屬性書寫順序/CSS屬性書寫順序.md#第二課再寫自身屬性建立元素盒子的大小與外觀底層) |
| 文字顯示不如預期 | 文本屬性與容器條件 | `color`、`font`、`text-align`、`white-space` | 文字設定應放在盒子條件之後，也可能受容器影響 | [第三課](../../notes/第02章_CSS屬性書寫順序/CSS屬性書寫順序.md#第三課接著寫文本屬性處理內容文字的呈現) |
| 只有圓角、陰影或游標不對 | 其他屬性區段 | `border-radius`、`box-shadow`、`cursor` | 這些通常是建立在版面、盒子與文字之後的細節效果 | [第四課](../../notes/第02章_CSS屬性書寫順序/CSS屬性書寫順序.md#第四課最後寫其他屬性處理裝飾互動與特殊內容) |
| 排序後畫面沒有變化 | 是否只是重新排列不同屬性 | 維護順序，不是屬性效果規則 | 本章排序多半改善可讀性，不一定改變畫面 | [誤解一](../../notes/第02章_CSS屬性書寫順序/CSS屬性書寫順序.md#誤解一屬性順序一定會改變畫面) |
| 不確定團隊是否必須完全照同一套順序 | 是否有一致的專案約定 | 團隊規範、排序一致性 | 來源提供的是建議順序，重點是同一專案內保持一致 | [誤解三](../../notes/第02章_CSS屬性書寫順序/CSS屬性書寫順序.md#誤解三所有團隊都必須使用完全相同順序) |
| 來源名稱看起來像完整 CSS 語法但不確定 | 來源是否提供完整上下文 | `break-word`、`background:liner-gradient...` | 筆記明確標示這些名稱缺少完整語法依據 | [誤解五](../../notes/第02章_CSS屬性書寫順序/CSS屬性書寫順序.md#誤解五來源中的每個名稱都能直接當成完整語法) |
