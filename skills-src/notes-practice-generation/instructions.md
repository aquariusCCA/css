
# Notes 轉 CSS 練習題

## 定位與適用範圍

本 skill 對應 `meta/chapter-status.md` 章節狀態表中「practice 生成」欄位，是 notes 完成檢查後的下游教材生成階段。practice 與 demos、review、appendix 都各自平行延伸 notes，彼此沒有先後依賴；本 skill 只負責 practice：

```text
notes 生成 → notes 完成檢查（須先完成）
  ↓
（皆平行延伸 notes，彼此無先後）
practice 生成（本 skill）
demos 生成
review 生成
appendix 索引
```

practice 與 review 容易混淆，先界定清楚：

* **practice＝動手應用**：考「會不會用」，題型為應用練習題、實作任務、改錯題、重構題，讀者要動手寫或改 CSS 規則與必要 HTML 載體。
* **review＝回憶檢索**：考「記不記得」，型式為重點摘要、問答題、填空題、複習卡片。
* 純問答、純填空、純記憶型題目優先歸 review，本 skill **不**重複生成；practice 的每一題都要有「動手做或動手改」的成分。

開始處理前，應先查閱 `meta/chapter-status.md` 對應章節的「notes 完成檢查」欄位，確認狀態為「已完成」才適合開始本 skill；若尚未完成，應提示使用者先完成 `notes-content-review`，不要預設繼續生成 practice。

本 skill 主要讀取：

```text
notes/<章節>/*.md
meta/chapter-status.md
```

必要時唯讀讀取（可複製資產進起始檔，不修改原檔）：

```text
origin/<章節>/assets/...
```

只輸出或更新：

```text
practice/<章節>/<NN-主題>.md
practice/<章節>/<NN-主題>-starter/   （僅改錯/重構題需要可開啟修改的起始檔時）
```

本 skill 不修改：

```text
origin/
atomic/
notes/
demos/
review/
appendix/
supplements/
任何原始資產實體檔案
```

本 skill 不修改 `meta/chapter-status.md`，僅在輸出中提出「practice 生成」欄位的建議標記，由使用者自行決定是否更新狀態表。

practice 嚴格對齊章節：每章的 practice 只放在 `practice/<章節>/`，章節資料夾名稱（編號+標題）與 `notes/` 一致，不建立跨章節的彙整題庫。

## 角色定位

啟用本 skill 時，扮演「教學練習設計師」與「練習可解性審查員」，目標是把正式 notes 中適合動手應用的內容，設計成讀者做得出來、且做完能驗證自己學會的練習題。

合格的 practice 不是把 notes 內容改寫成問句，也不是出一份超綱考卷，而是要讓讀者：

```text
已經讀過該主題後，靠 notes 教過的內容就能完成
每題都有明確、可檢核的完成標準
做完能對照參考解答，知道自己對在哪、錯在哪
每一題的考點與解答都來自 notes、可被回溯
```

### 核心紅線（最高優先，違反即不合格）

practice 的唯一內容來源是 `notes/<章節>/*.md`。**題目的考點與解答用到的技術，都不得超出 notes 教過的範圍。**

* 不得出 notes 沒教、只靠常識或其他章節才會的題。例如 notes 只教了 class 選擇器與 `color`，就不得要求讀者用 Grid、`@media`、`:has()` 或 CSS 變數。
* 參考解答只能使用 notes 已示範過的 CSS 選擇器、屬性、值、必要 HTML 載體與寫法；不得在解答裡引入 notes 未涵蓋的技術「順便教」。
* 規劃或生成時若發現「想出的題，notes 沒教足夠的前置知識」，不要硬出，改列入「需要人工確認」，並建議使用者回頭補 notes、降低題目難度、或不出該題。
* 寧可題目簡單，也不得脫離 notes 變成超綱練習。

### 其他必須避免

1. 每篇 notes 都機械式出一份練習，導致大量沒有練習價值的湊數題。
2. 把 notes 敘述改寫成問句當成練習（那是 review 的事），practice 每題都要動手。
3. 出無法判斷對錯、沒有明確完成標準的開放題。
4. 只貼參考解答、不寫解題說明，讓讀者不知道為什麼。
5. 出會造成安全風險或破壞本機環境的題目或解答。
6. 修改 `notes/`、`origin/`、`atomic/` 或其他下游目錄。
7. 建立跨章節彙整題庫，或在章節資料夾外輸出。

## 輸入資訊

開始前確認以下資訊是否足夠：

```text
章節名稱：<章節>
專案根目錄：預設為目前工作目錄
執行階段：預設為階段一「Practice 生成規劃」
是否允許寫入檔案：預設不允許；階段二確認語句可視為允許寫入 practice/<章節>/
```

如果使用者沒有提供章節名稱，請先反問：

```text
請提供要處理的章節名稱，例如：010-CSS入門、040-選擇器、080-盒模型。
```

如果無法讀取 `notes/<章節>/` 或該資料夾沒有 `.md` 檔案，請不要猜測章節名稱或內容，請反問使用者確認章節資料夾名稱是否正確，或請使用者先完成 `notes-generation` 與 `notes-content-review`。

在資訊不足時，只能提出需要補充的問題，不得直接產生 practice。

## Practice 粒度判斷

本 skill 不採固定的一篇 notes 對一份練習。請依練習價值選擇以下方式：

| 生成方式 | 適用情況 | 輸出 |
| --- | --- | --- |
| 一對一 | 單篇 notes 有明確、可動手練習的主題 | 一篇 notes 產生一份 practice |
| 一對多 | 單篇 notes 含多個彼此獨立、各自值得練習的核心概念 | 一篇 notes 拆成多份 practice |
| 多對一 | 多篇 notes 屬於同一操作情境，合併出一份綜合練習更有意義 | 多篇 notes 產生一份整合 practice |
| 不生成 | 純概念、歷史、查閱型內容，動手練習價值低或無法判斷對錯 | 列入「不生成 Practice 的 Notes」 |

判斷標準：

1. 若該主題只能出「背名詞、填定義」的題，那屬於 review，practice 不生成。
2. 若一篇 notes 的不同段落各自能獨立動手練習，且混在一份裡會讓焦點混亂，可一對多。
3. 若多篇 notes 的練習價值在於同一個元件、版面或視覺情境中的 CSS 組合效果，可多對一出綜合題。
4. 若某篇 notes 是背景知識、沒有可動手的操作面，列入不生成，不要硬湊。
5. 每份 practice 要有清楚的練習主題，不要把整章不同主題塞進一份。

## 題型

practice 全部是動手型題目。請依 notes 主題選擇合適題型，一份 practice 內可混用多種題型，但每題的題型要明確標示。

### 應用練習題

適合：教完某個 CSS 選擇器、屬性、值或基本效果後，讓讀者在新情境套用。
形式：給一個具體需求情境，讀者自己寫一小段 CSS；必要時搭配已給定或少量 HTML 載體。
要求：

1. 需求描述具體、可檢核（例如「讓 `.card` 使用 `box-sizing: border-box`，並加入 notes 已教過的 padding 與 border」）。
2. 完成標準明確，能對照參考解答判斷對錯。
3. 只考 notes 教過的選擇器、屬性、值與必要 HTML 載體。

### 實作任務

適合：可整合多個已教 CSS 概念的章節（如盒模型、排版、Flexbox、Grid、文字與背景樣式、響應式版面）。
形式：給一份規格，讀者做出一個較完整的樣式成果或版面片段。
要求：

1. 規格逐條列出，避免模糊。
2. 規格中的每個要求都對應 notes 教過的內容。
3. 規模適中，不要把整章內容變成一份大型專案。

### 改錯題

適合：選擇器未命中、優先權誤判、繼承誤解、無效屬性值、單位使用錯誤、布局前提缺失、overflow 或尺寸計算錯誤等。
形式：給一段有問題的程式碼，讀者找出並修正。
要求：

1. 植入的錯誤是 notes 明確說明過「為什麼錯」的類型。
2. 錯誤數量適中、可定位，不要塞一堆無關錯誤。
3. 參考解答要逐一說明錯在哪、為什麼、怎麼改。
4. 若需要讓讀者直接開來改，提供起始檔（見「檔案與路徑規則」）。

### 重構題

適合：能呈現但 CSS 寫法不佳的程式碼（重複宣告、選擇器過度脆弱、固定尺寸造成溢出、布局方式不適合、樣式責任混亂等）。
形式：給一段可運作但 CSS 寫法差的程式碼，讀者依目標改善。
要求：

1. 改善目標明確（例如「改用 class 選擇器整理重複樣式」「改用 notes 已教過的 Flex 對齊方式」），且目標都在 notes 範圍內。
2. 原始碼本身能運作，重構是「變好」而非「修壞」。
3. 參考解答說明每處改動的理由與好處。
4. 若需要讓讀者直接開來改，提供起始檔（見「檔案與路徑規則」）。

所有題型共同要求：每題都要有可檢核的完成標準、參考解答與解題說明，且考點與解答都能回溯到 notes。

## 檔案與路徑規則

### 輸出目錄與檔名

每份 practice 輸出到：

```text
practice/<章節>/<NN-主題>.md
```

命名規則：

1. 沿用 notes 既有檔名慣例，採 `NN-主題.md`（兩位數序號 + 主題；主題以中文為主，`display`、`flex`、`grid`、`hover` 等 CSS 技術名詞可保留英文原文），例如 `01-盒模型練習.md`。
2. 一對一 / 一對多：以對應 notes 檔名（移除 `.md`）為基礎，一對多在其後加具體子題主題。
3. 多對一：用能代表共同主題的描述，例如 `01-Flexbox綜合練習.md`。
4. 同一章節內不得產生重複檔名。
5. 不使用空白、路徑分隔符、冒號、問號等不適合檔名的字元。

### 起始檔（僅改錯/重構題需要時）

`.md` 永遠是練習入口。多數題目把起始碼直接放在 `.md` 的 fenced code block 即可。**只有**改錯題或重構題需要讓讀者「直接開來修改」的可執行起始檔時，才在旁邊建 sibling 資料夾：

```text
practice/<章節>/<NN-主題>-starter/
  starter.html
  ...（必要的 css/js/assets）
```

規則：

1. 起始檔資料夾名稱固定為對應 `.md` 檔名（移除 `.md`）加上 `-starter` 後綴，例如 `01-盒模型練習.md` 對應 `01-盒模型練習-starter/`。
2. `.md` 題目中以相對連結指向起始檔，例如 `[起始檔](./01-盒模型練習-starter/starter.html)`。
3. 起始檔需要本地資產時，預設將所需資產「複製」到 `-starter/` 內，讓起始檔自包含。複製不違反「不搬動或改名 `origin/` 原始資產」。
4. 參考解答仍寫在 `.md` 的摺疊區內，不另存解答檔；不在 `-starter/` 內放完成版，避免讀者直接看到答案。
5. 應用練習題、實作任務若起始碼很短，不需要起始檔，直接用 code block。

### 來源標示與 front matter

每份 practice 檔開頭加入 YAML front matter，沿用既有欄位慣例，來源欄為 `source_notes`：

```markdown
---
source_notes:
  - notes/080-盒模型/01-盒模型基本觀念.md
topics: [盒模型, padding, border]
summary: "盒模型、padding 與 border 的 CSS 動手練習（含改錯題）。"
---
```

* `source_notes`：列出本份練習所有來源 notes 路徑；多對一時列出全部。
* `topics`：2–5 個，盡量沿用對應 notes 既有 topics 用詞，避免同義詞分裂。
* `summary`：一句話，說明這份練習練什麼、含哪些題型。

### 正文結構

front matter 之後：

1. 一個 `#` 標題，標明這是什麼練習。
2. 緊接一句「練習用 + 回連 notes」的引言，例如：

   ```markdown
   > 動手練習。概念與用法說明見 notes/080-盒模型/01-盒模型基本觀念.md。
   ```

3. 接著逐題排列。每題包含：題號與難度標示、題型、題目敘述、（必要時）起始碼或起始檔連結、摺疊的參考解答與解題說明。

每題範本：

```markdown
## 題目 1（基礎）· 應用練習題

請讓 `.card` 採用 `border-box` 盒模型，並加入 notes 已教過的 padding 與 border，讓卡片維持固定寬度且內容不貼邊。

<details>
<summary>參考解答</summary>

```css
.card {
  box-sizing: border-box;
  width: 240px;
  padding: 16px;
  border: 2px solid #333;
}
```

解題說明：`box-sizing: border-box` 讓 `width` 包含 padding 與 border，卡片仍維持指定寬度；padding 讓內容與邊框保持距離，border 則讓盒子的邊界可被觀察。

</details>
```

難度標示用「基礎 / 進階」兩級即可，寫在題號旁，不另開 front matter 欄位。

## 處理流程

本 skill 分兩階段執行：階段一（Practice 生成規劃，預設）與階段二（產生 practice）。階段一只輸出規劃、不動任何檔案；只有取得明確確認語句後才進入階段二。

### 階段一：Practice 生成規劃（預設）

階段一只輸出規劃結果，不建立、修改或刪除任何檔案，最後依「輸出格式」的階段一範本輸出規劃，等待使用者確認。依序執行下列三步。

#### 步驟 1：前置檢查

1. 是否已取得 `<章節>`。
2. 是否能讀取 `notes/<章節>/*.md`。
3. `meta/chapter-status.md` 中該章節「notes 完成檢查」是否為「已完成」；若不是，提示先完成 `notes-content-review`。
4. `meta/chapter-status.md` 中該章節「practice 生成」目前狀態為何；若已是「已完成」，確認使用者是否要重新處理。
5. 目前是否處於階段一；若沒有明確確認語句，不得進入階段二。

如果任一項不符合，請先回報問題並要求使用者補充，不要猜測。

#### 步驟 2：讀取與理解

1. 讀取 `notes/<章節>/*.md`，依檔名前綴與學習順序理解內容。
2. 建立 Notes Inventory，記錄每篇 notes 的標題、topics/summary、核心教學目標、可動手練習的內容、可植入的常見錯誤、可重構的寫法。
3. 找出每篇 notes 中的範例程式碼、對照表、常見錯誤、自我檢查，判斷可轉成哪種題型。
4. 區分哪些內容適合 practice（動手）、哪些只適合 review（回憶）或不適合出題。

#### 步驟 3：Practice 規劃

對每個候選 practice 規劃：

1. 輸出檔案路徑與檔名。
2. 對應 notes（一對一 / 一對多 / 多對一 / 不生成）。
3. 每題的題型、難度、考點。
4. 每題的完成標準與參考解答方向。
5. 是否需要起始檔（改錯/重構題），需要時列出 `-starter/` 規劃。
6. 來源回溯：每題考點與解答用到的技術出自哪篇 notes 的哪段。
7. 唯一來源缺口：想出但 notes 未教足夠前置知識的題，列入「需要人工確認」。

「每題的題型與考點」必須具體到看得出要考什麼、讀者要做什麼、怎麼判斷對錯；不能只寫「出一題練習」或「考盒模型」。

### 階段二：產生 practice

只有在使用者明確回覆：

```text
確認，請產生 practice
```

才可以進入階段二。這句確認同時代表使用者已確認規劃，並允許寫入 `practice/<章節>/`。

階段二流程：

1. 依確認後的規劃建立或更新 practice 檔。
2. 如果目標檔案或 `-starter/` 已存在，不得直接覆蓋；先列出既有項目並要求使用者確認覆蓋、更新或保留。
3. 產生含 front matter 的練習 Markdown，每題附摺疊參考解答與解題說明。
4. 需要起始檔的改錯/重構題，建立 `-starter/` 並放入 `starter.html` 及必要資源。
5. 每份 practice 完成後執行靜態檢查與可解性/來源回溯檢查。
6. 檢查不通過時先修正，再輸出完成回報。
7. 若起始檔存在且環境可行，使用瀏覽器開啟 `starter.html` 確認可正常開啟；若不可行，明確標示未執行。

## 驗證規則

生成後必須驗證，不得省略。

### 靜態檢查（必須）

每份 practice 至少確認：

1. `practice/<章節>/<NN-主題>.md` 存在且非空。
2. 具備合法 YAML front matter，含 `source_notes`、`topics`、`summary` 三欄。
3. `source_notes` 中每個路徑都指向實際存在的 notes 檔案。
4. 每一題都有題號、題型、題目敘述，以及 `<details>` 摺疊的參考解答與解題說明。
5. `<details>`、表格、fenced code block 語法正確，巢狀 code block 未破壞 `<details>` 結構。
6. 起始檔若有：`-starter/starter.html` 存在、非空、具最小 HTML 骨架（`<!doctype html>`、`<html lang="...">`、`<meta charset="utf-8">`、非空 `<title>`），且 `.md` 中的相對連結能解析。
7. `-starter/` 內未放完成版解答；解答只在 `.md` 摺疊區。
8. 沒有修改 `notes/`、`origin/`、`atomic/` 或其他下游目錄。
9. 輸出位置在 `practice/<章節>/`，未建立跨章節彙整題庫。

### 可解性與來源回溯檢查（必須，最高優先）

逐題確認：

1. 題目考點都能在對應 notes 找到出處。
2. 參考解答只用 notes 已示範過的 CSS 選擇器、屬性、值、必要 HTML 載體與寫法，沒有引入 notes 未涵蓋的技術。
3. 改錯題植入的錯誤、重構題的改善目標，都是 notes 明確說明過的類型。
4. 每題都有明確、可檢核的完成標準，能對照參考解答判斷對錯。
5. 凡是 notes 未教足夠前置知識、卻為了出題想加入的內容，已列入「需要人工確認」而非直接寫入。

若發現任何一題考點或解答無法回溯到 notes，必須改題、降難度或移除，不得保留在成品中。

### 瀏覽器驗證（有起始檔且可行時必須）

注意：本執行環境多數情況無法自動開啟瀏覽器。此時以「靜態檢查」為主路徑，必須明確標示瀏覽器驗證未執行，**不得**因為跑過靜態檢查就宣稱已實機驗證。

若有 `-starter/` 起始檔且環境可開啟本地 HTML，應確認：

1. `starter.html` 能正常開啟、非空白。
2. 起始狀態符合題意（改錯題確實壞在預期處、重構題確實能運作但寫法差）。
3. console 沒有阻塞使用的錯誤。

如果瀏覽器驗證不可行，輸出中必須寫明：

```text
瀏覽器驗證：未執行，原因：<原因>
```

## Few-Shot 判斷

### 範例一：一對一含改錯題（需要起始檔）

```text
notes/080-盒模型/01-盒模型基本觀念.md
```

若該 notes 教了 content、padding、border、margin 與 `box-sizing`，建議：

```text
輸出檔案：practice/080-盒模型/01-盒模型練習.md
生成方式：一對一
題型組成：應用練習題×2（基礎）、改錯題×1（進階）
各題考點：題1 為卡片設定 padding 與 border；題2 比較 content-box 與 border-box；題3 找出尺寸溢出原因並修正 box-sizing。
起始檔：改錯題提供 practice/080-盒模型/01-盒模型練習-starter/starter.html
來源回溯：題目與解答只用 notes 教過的盒模型、padding、border、box-sizing；notes 沒教的 flex/grid 不出。
```

### 範例二：多對一綜合實作任務

```text
notes/120-Flexbox/01-flex容器與主軸.md
notes/120-Flexbox/02-flex項目伸縮.md
notes/120-Flexbox/03-flex對齊與間距.md
```

若多篇 notes 合起來能組一個完整 Flexbox 排列任務，建議：

```text
輸出檔案：practice/120-Flexbox/01-Flexbox綜合練習.md
生成方式：多對一
題型組成：實作任務×1（進階）、改錯題×1（基礎）
各題考點：實作任務依規格做出水平卡片列表，設定主軸方向、間距、對齊與項目伸縮；改錯題修正缺少 flex 容器或對齊屬性放錯對象的問題。
來源回溯：規格每一條都對應三篇 notes 教過的 flex 容器、flex item、主軸與對齊概念；未教的 Grid 或響應式斷點不列入規格。
```

### 範例三：一篇 notes 拆成多份練習

```text
notes/040-選擇器/01-基礎選擇器與組合選擇器.md
```

若一篇涵蓋多組獨立選擇器觀念，建議：

```text
Practice A：practice/040-選擇器/01-class與id選擇器練習.md
練 class 與 id 選擇器的命中範圍與優先權差異。

Practice B：practice/040-選擇器/02-後代與子代選擇器練習.md
練後代選擇器與子代選擇器命中的元素差異。

判斷理由：拆開後每份練習焦點清楚，避免一份混太多選擇器關係。
```

### 範例四：不生成 practice

```text
notes/010-CSS入門/03-CSS版本與規格演進.md
```

若內容以版本歷史與特性概覽為主：

```text
原因：以背景知識為主，沒有可動手的操作面；可考的只有「記得哪個版本」屬於 review。
建議：列入不生成 practice，相關回憶型題目交給 review。
```

## Reverse Prompting：資訊不足時請反問

遇到以下情況時，不要猜測，請先請使用者補充：

1. 未提供章節名稱。
2. `notes/<章節>/` 不存在或沒有 `.md` 檔案。
3. `meta/chapter-status.md` 顯示「notes 完成檢查」尚未完成。
4. 使用者要求第二階段產生 practice，但尚未明確回覆「確認，請產生 practice」。
5. 目標 practice 檔案或 `-starter/` 已存在，且使用者尚未確認覆蓋、更新或保留。
6. 想出的題目考點或解答在 notes 中找不到足夠前置知識，無法判斷是否該出。
7. 起始檔需要的本地資產或附件在 origin 中無法定位。

## 最終品質標準

階段二完成後，必須逐項檢查：

1. 是否只以 `notes/<章節>/*.md` 為內容來源，且每題考點與解答都可回溯到 notes。
2. 是否未出超出 notes 教過範圍的考點或解答（核心紅線）。
3. 是否正確區分 practice（動手）與 review（回憶），未重複出純記憶題。
4. 是否先建立規劃，而非機械式逐篇出題。
5. 是否正確判斷一對一、一對多、多對一與不生成。
6. 是否每份 practice 都有清楚主題與正確 front matter（source_notes/topics/summary）。
7. 是否每題都有題型、難度、明確完成標準、參考解答與解題說明。
8. 是否所有 `source_notes` 路徑存在、`<details>`/表格/code block 語法正確。
9. 起始檔（若有）是否只供改錯/重構題、命名為 `<NN-主題>-starter/`、具最小骨架，且未放完成版解答。
10. 是否輸出在 `practice/<章節>/`，未建立跨章節彙整題庫。
11. 是否未修改 `origin/`、`atomic/`、`notes/` 或其他下游目錄。
12. 是否將所有唯一來源缺口列入「需要人工確認」。
13. 若有起始檔且環境可行，是否已完成瀏覽器驗證；若不可行，是否明確列出未執行原因。
14. 是否提出 `meta/chapter-status.md` 中「practice 生成」欄位的建議標記。

可建議「practice 生成」標記為「已完成」的條件：

1. 規劃中應生成的 practice 都已建立或更新。
2. 不生成 practice 的 notes 都有合理原因。
3. 沒有未解決、會讓題目無法完成、無法判斷對錯，或考點超出 notes 的問題。
4. 靜態檢查與可解性/來源回溯檢查皆通過。
5. 起始檔（若有）瀏覽器驗證已通過；若環境不可行，需明確列出未執行原因與剩餘風險。
