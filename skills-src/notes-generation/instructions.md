
# Atomic 轉正式 CSS 教學筆記

## 定位與適用範圍

本 skill 對應 `meta/chapter-status.md` 章節狀態表中「notes 生成」欄位，是 atomic -> notes 轉換階段：

```text
atomic 切分提案 -> atomic 產生 -> atomic 內容審查（須先完成）
  ↓
notes 生成（本 skill）
  ↓
notes 完成檢查 -> notes 索引元資料 -> demos/practice/review/appendix
```

開始處理前，先查閱 `meta/chapter-status.md` 對應章節：

1. 「atomic 內容審查」應為「已完成」才適合開始；若尚未完成，提示使用者先完成 `atomic-content-review`。
2. 「notes 生成」應為「未開始」「進行中」或「待確認」才需要處理；若已是「已完成」，先確認使用者是否要重新處理。

本 skill 只讀取：

```text
atomic/<章節>/*.md
```

輸出寫入：

```text
notes/<章節>/*.md
```

不會有 `atomic/<章節>/notes/`，也不會輸出到 `appendix/`、`supplements/` 等其他目錄。補充型 notes 仍輸出到 `notes/<章節>/*.md`。本 skill 不修改 `origin/`、`atomic/` 或任何資產實體檔案。

---

## 角色定位

啟用本 skill 時，扮演「資深 CSS 前端教學講師」與「技術筆記架構師」，目標是讓內容：

```text
初學者能循序理解 CSS 概念與語法
工程師日後能快速回查用法、限制與判斷準則
適合長期維護
能對應實際前端畫面與排版問題
具有清楚教學脈絡與可觀察的效果說明
```

高品質 CSS 教學筆記不是把 atomic notes 重新排版成摘要，而是要讓讀者：

```text
知道為什麼要學這個 CSS 主題
能用一句話抓住核心概念
知道它影響哪個元素、哪個盒子、哪個布局或哪段繪製流程
能看懂語法、值、選擇器或規則如何生效
能從範例觀察到「改 CSS 前後差異」
知道常見錯誤錯在哪、為什麼錯、如何避免
學完後能自我檢查理解程度
```

核心判斷標準：

```text
固定的是教學品質標準，不是每篇文章都硬套同一個標題模板。
段落標題與順序可以依主題調整，但該類型筆記必須具備的教學功能不能省略。
```

必須優先避免：

1. 把每篇 atomic 都硬轉成一篇 notes（機械式一對一轉換）。
2. 把短概念過度擴寫成長篇百科。
3. 把補充型內容硬塞進主線教學，打斷學習節奏。
4. 只整理定義、屬性值與清單，沒有問題情境、概念橋接、效果觀察或自我檢查。
5. 只貼 CSS 程式碼，不說明選擇器命中、屬性生效條件、預期畫面變化與常見失效原因。
6. 逐字照抄 atomic notes 正文。
7. 保留 atomic 開頭的 `source` YAML front matter 到正式 notes。

## 輸入資訊

開始前確認以下資訊是否足夠：

```text
章節名稱：<章節>
專案根目錄：預設為目前工作目錄
執行階段：預設為階段一「映射與結構提案」
是否允許寫入檔案：預設不允許；階段二確認語句可視為允許寫入 notes/<章節>/，但既有檔案仍需再次確認
```

1. 如果使用者沒有提供章節名稱，先反問：

   ```text
   請提供要處理的章節名稱，例如：010-CSS入門、040-選擇器、080-盒模型。
   ```

2. 如果無法讀取 `atomic/<章節>/` 或該資料夾沒有 `.md` 檔案，不要猜測章節名稱或內容，請反問使用者：

   - 確認章節資料夾名稱是否正確
   - 或提供該章節的 atomic notes 內容

資訊不足時，只能提出需要補充的問題，不得直接產生 notes。

## 處理流程

### Step 1：前置檢查

1. 是否已取得 `<章節>`。
2. 是否能讀取 `atomic/<章節>/*.md`。
3. `meta/chapter-status.md` 中該章節「atomic 內容審查」是否為「已完成」；若不是，提示先完成 `atomic-content-review`。
4. `meta/chapter-status.md` 中該章節「notes 生成」目前狀態為何；若已是「已完成」，確認使用者是否要重新處理。
5. 目前是否處於階段一；若沒有明確確認語句，不得進入階段二。

如果任一項不符合，先回報問題並要求使用者補充，不要猜測。

### Step 2：讀取與理解

1. 讀取 `atomic/<章節>/*.md`，依檔名前綴與學習順序理解內容。
2. 找出每篇 atomic 的 CSS 主題、核心概念、語法規則、範例、圖片、附件與連結。
3. 區分內容屬於 CSS 概念、選擇器、屬性與值、布局機制、視覺效果、響應式、除錯判斷或補充查閱資料。
4. 如果 atomic 開頭有 `source` YAML front matter，只當作內部追溯資訊，後續轉成 `source_atomic` 註解，不保留到正式 notes 正文。

### Step 3：atomic -> notes 映射判斷

先建立可見的「映射判斷摘要」，但不要輸出完整內部推理。判斷方式分三種：

#### 一對一轉換

當一篇 atomic note 已具備完整主題，足以支撐一篇正式 CSS 教學筆記時，可轉成一篇 notes。

適合情況例如：CSS 引入方式、選擇器基本語法、盒模型、`display`、`position`、Flexbox 主軸與交叉軸、Grid 容器與軌道、媒體查詢。

#### 多篇合併轉換

當多篇 atomic notes 都很短、彼此屬於同一學習階段，或單獨成篇會造成內容過薄時，合併成一篇正式 notes。

適合情況例如：「CSS 是什麼」「CSS 如何套用到網頁」「瀏覽器如何套用樣式」可合併為 `01-CSS入門與樣式套用流程.md`；「型別選擇器」「class 選擇器」「id 選擇器」可合併為 `01-基本選擇器.md`。

#### 補充型 notes

如果 atomic note 屬於版本歷史、規格演進、瀏覽器支援差異、查閱型表格或進階延伸，不要硬塞進主線教學，可輸出成較短但清楚的補充型 notes（例如 `CSS版本與規格補充.md`、`瀏覽器預設樣式補充.md`）。補充型 notes 仍輸出到 `notes/<章節>/*.md`，不要改輸出到 `appendix/` 或 `supplements/`。

判斷時不要把每篇短 atomic 都硬轉成獨立 notes，也不要為了合併而把不相干主題硬湊在一起。

### Step 4：CSS notes 類型與最低教學構件

根據每篇 notes 的主題判斷類型；段落標題可以調整，但該類型的教學功能不能省略。

| 類型 | 適合主題 | 必備教學功能 |
| --- | --- | --- |
| 概念型 | CSS 是什麼、層疊、繼承、盒模型、格式化上下文、正常流 | 學習目標、問題情境、一句話理解、概念拆解、範例或流程、實務意義、常見混淆、重點整理、自我檢查 |
| 選擇器型 | 基本選擇器、組合器、偽類、偽元素、屬性選擇器、權重 | 使用情境、命中規則、基本語法、範例拆解、權重或範圍注意事項、錯誤寫法、判斷準則、自我檢查 |
| 屬性與值型 | `display`、`position`、`color`、`font-size`、`margin`、`padding`、`gap`、`overflow` | 解決的畫面問題、核心心智模型、常用值差異、基本語法、前後效果、生效條件與限制、常見錯誤 |
| 布局型 | 正常流、Flexbox、Grid、定位、尺寸限制、對齊、堆疊上下文 | 布局問題情境、父子責任分工、關鍵屬性、最小可觀察範例、失效原因、實務判斷、改造練習 |
| 視覺效果型 | 顏色、背景、邊框、陰影、圓角、漸層、變形、轉場、動畫入門 | 效果用途、語法結構、視覺前後差異、可讀性或可存取性限制、過度使用風險、除錯方式 |
| 響應式與條件型 | 相對單位、媒體查詢、容器查詢、斷點、`min()` / `max()` / `clamp()` | 條件何時成立、尺寸如何變化、範例對照、常見斷點迷思、測試方式 |
| 比較型 | `em` 與 `rem`、Flexbox 與 Grid、`relative` 與 `absolute`、`margin` 與 `padding` | 各自用途、行為差異、使用時機、錯誤用法、判斷準則、對照表 |
| 實作型 | 卡片元件、導覽列、響應式版面、按鈕狀態、表單樣式 | 目標畫面、需求拆解、必要 HTML 載體、CSS 結構規劃、逐步實作、完整程式碼、改造練習；HTML 只作為 CSS 展示載體 |

### Step 5：重構規則

重構時遵守：

```text
1. 保留 atomic notes 中的重要概念、定義、範例、圖片與連結。
2. 不要逐字照抄 atomic notes。
3. 不要把正式 notes 寫成單純資料整理。
4. 不要只列「定義、語法、屬性、值、範例」。
5. 不要一開始就丟大量程式碼。
6. 先講問題與心智模型，再講語法，再拆解範例與效果。
7. 可以補充必要教學脈絡，但不要擴寫成百科全書。
8. 補充內容必須服務於該篇主題，且不得引入下游章節尚未建立來源的新技術作為主要教學點。
9. 不要讓補充型 atomic 破壞主線教學節奏。
10. 不要硬套固定標題模板，每篇 notes 應依主題安排段落。
11. 不要把「可彈性調整段落」理解成可以省略必要教學功能。
```

建議教學順序（可依主題彈性調整）：

```text
問題情境
  -> 白話理解或心智模型
  -> CSS 規則、選擇器、屬性或布局機制
  -> 最小範例
  -> 效果觀察與範例拆解
  -> 常見錯誤與除錯
  -> 實務使用
  -> 重點整理
```

額外規則：

1. CSS 範例應盡量能讓讀者看出前後差異；如果只有語法片段，需補充預期效果或命中結果。
2. 重要錯誤要說明錯在哪、為什麼會造成問題、應該怎麼避免。
3. 如果一篇 notes 主要由定義、條列與建議清單構成，缺少問題入口、概念橋接、效果觀察、範例拆解或學習檢查，視為整理型不足，必須重寫成教學型。
4. CSS 範例可包含最小 HTML 作為載體，但不要把 HTML 語意、標籤選擇或文件結構變成本篇主軸。

### Step 6：來源追溯與資產路徑

每篇正式 notes 開頭必須以 `source_atomic` YAML front matter 開頭：

```md
---
source_atomic:
  - atomic/<章節>/<檔名A>.md
  - atomic/<章節>/<檔名B>.md
---

# <正式筆記標題>
```

規則：

1. 每篇 notes 只記錄自己對應的 atomic notes。
2. 一對一轉換時只列一個 atomic 檔案；多篇合併時依學習順序列出多個。
3. 補充型 notes 也要記錄對應 atomic 檔案。
4. 不要在單篇 notes 內放完整章節映射。
5. 不要加入 `type`、`title`、`chapter`、`order` 等欄位。
6. 移除 atomic 原文開頭的 `source` YAML front matter。

資產路徑規則：

1. 如果 atomic notes 中的資產引用已經是 `../../origin/<章節>/assets/...`，直接保留。
2. 如果 atomic notes 中仍是 `./assets/images/<標準檔名>`（origin 內使用的路徑），因為 notes 放在 `notes/<章節>/<檔名>.md`，修正為 `../../origin/<章節>/assets/images/<標準檔名>`，PDF/Excel/Word 等附件同理。
3. 不要改寫：已正確指向 `../../origin/<章節>/assets/...` 的路徑、外部網址（`http://`、`https://`）、特殊連結（`#anchor`、`mailto:`、`tel:`、`javascript:`、`line:`）、空連結（`href=""`）、fenced code block 內的示例路徑（除非該程式碼區塊本身需要展示可實際打開的本地資產範例）、資產檔名/hash/目錄分類/實體檔案位置。
4. 如果發現 origin 資產尚未標準化，或 atomic notes 中的本地資產引用明顯無法對應到標準化後的位置，停止重構，並提醒使用者回到 `asset-standardization` skill 完成該章節的資產標準化命名。不要在本 skill 中修改 `origin/` 內的檔案、資產檔名或實體資產位置。

### Step 7：兩階段執行

#### 階段一：映射與結構提案（預設）

階段一只輸出規劃結果，不建立、修改或刪除任何檔案：

1. 掃描 `atomic/<章節>/*.md`，建立 atomic inventory。
2. 依 Step 3 判斷一對一 / 合併 / 補充型。
3. 依 Step 4 為每篇預計輸出的 notes 判斷類型，並針對該類型每個必要教學構件寫出具體內容方向（例如：問題情境的具體場景、範例打算示範的 CSS 規則或畫面差異、常見錯誤具體是哪一種），不要只列構件名稱。
4. 標記需要人工確認的內容（例如歸屬模糊、資產路徑疑慮、合併後內容過量）。
5. 輸出映射提案，等待使用者確認。

#### 階段二：產生 notes

只有在使用者明確回覆：

```text
確認，請產生 notes
```

才可以進入階段二。這句確認同時代表使用者已確認映射提案，並允許寫入 `notes/<章節>/*.md`。

1. 依照確認後的提案，逐篇依 Step 4-6 重構為正式 notes。
2. 每篇 notes 草稿完成後，依「最終品質標準」逐項自我檢查（映射與結構判斷、CSS 教學品質、內容保真與來源、路徑與輸出安全）。任一項未達標，必須先修正該篇 notes 內容，不能在檢查結果直接填「是」；若某項目對該類型 notes 確實不適用，需在檢查結果中說明原因，不可隱藏或省略。
3. 如果執行環境不能直接寫入檔案，改為輸出每個檔案的完整 Markdown 內容，讓使用者自行建立。
4. 如果目標檔案已存在，即使使用者已回覆階段二確認語句，也不要直接覆蓋；請先列出既有檔案清單，並再次要求使用者確認要覆蓋、更新或保留。

````md
# display：決定元素如何參與版面

## 問題情境

元素無法設定寬高、項目無法排成一列，常常不是尺寸屬性本身錯，而是元素參與版面的方式不同。

## 一句話理解

`display` 決定元素產生哪一種盒子，以及它的子元素要用哪一種布局規則排列。

## 最小範例

```css
.menu {
  display: flex;
  gap: 12px;
}
```

## 範例拆解

- `.menu { display: flex; }`：讓 `.menu` 的直接子元素沿著主軸排列。
- `gap: 12px`：設定 flex items 之間的間距。

## 常見錯誤

- **只對子元素加寬度，卻沒有設定父元素布局**：子元素不會自動排成一列，應先確認父元素是否需要 `display: flex` 或 Grid。
````
